# AdOps Copilot 投放归因排障助手项目复盘与面试追问 v1.0

本文档用于面试准备，不是对外 PRD。目标是把 PRD 背后的 Why、How、Trade-off、Corner Case 和可被追问的问题沉淀下来，帮助面试时讲得像真正做过 AI 产品落地的人。

重要边界：当前项目材料是产品方案与作品集材料，不应把未验证指标说成真实业务结果。涉及“节省 30% 时间”“采纳率 60%”等内容时，只能表达为目标、灰度门槛或测算方法，除非后续补充真实灰度数据。

# 1. 项目一句话

AdOps Copilot 投放归因排障助手面向广告投放运营、AM 和技术支持，解决投放异常与平台/MMP/客户后台数据不一致问题。它不是开放式聊天机器人，而是一个受权限、证据、工具和评测约束的内部排障工作流中枢。

面试表达可以这样说：

```text
我把这个项目定位成“证据驱动的投放归因排障助手”，而不是泛泛的广告运营问答。
原因是广告投放和归因问题往往涉及多个系统和口径，如果只靠大模型回答，很容易出现幻觉、越权和口径不一致。
所以我把方案拆成总控路由、场景智能体、RAG 知识检索、只读数据工具、证据对象、人工确认和 Badcase 回流几层，让 AI 负责拆解和组织诊断，让确定性工具负责数据和口径，让人工保留最终确认权。
```

# 2. 为什么先做投放归因排障

## 2.1 选择这个切入点的原因

| 判断维度 | 选择理由 | 面试可讲重点 |
| --- | --- | --- |
| 高频 | 投放异常、转化下降、MMP 数据不一致是一线常见问题 | 能积累足够 Badcase 和评测样本 |
| 数据可验证 | 问题通常能落到指标、账户状态、归因窗口、postback 状态 | 适合做证据型 AI，而不是纯生成 |
| 风险可控 | 当前阶段只读查询，不自动改预算、出价或账户配置 | 更容易通过灰度和权限审查 |
| 业务价值清晰 | 可以衡量处理时长、升级率、采纳率、上下文完整率 | 比“提升体验”更容易量化 |
| 可扩展 | 后续能扩到 SDK/API 排障、素材审核、客户回复 | 第一阶段先把底座跑通 |

## 2.2 为什么不一开始做全能 Copilot

全能 Copilot 看起来范围更大，但落地风险高：问题类型太多、权限边界复杂、评测集难建立、用户预期不可控。面试时可以强调：我选择先聚焦一个高频、证据明确、权限可控的场景，把 RAG、工具调用、证据对象、评测和 Badcase 闭环做扎实，再逐步扩展。

# 3. 端到端链路怎么讲

以“客户反馈 MMP installs 比平台报表低 28%”为例：

1. 用户输入问题，系统先识别 campaign、app、event、MMP、时间窗和语言。
2. 总控智能体做意图识别、权限校验和场景路由，判断这是归因数据不一致核对。
3. 归因核对智能体拆解检查项：时区、归因窗口、事件映射、去重、postback 延迟、MMP 刷新、渠道映射。
4. 系统调用 RAG 检索归因口径和 SOP，同时调用平台报表、MMP 报表和 postback 摘要工具。
5. 所有返回结果转成 evidence object，记录来源、时间、权限、Owner、置信度。
6. 场景智能体基于证据输出诊断：可能原因排序、已排除项、下一步建议、需要人工确认的点。
7. 总控做安全交付，避免输出无证据强结论或客户可直接发送话术。
8. 用户反馈采纳、无效、继续追问或升级工单，反馈进入 Badcase 和评测集。

面试关键词：不是“LLM 直接回答”，而是“意图路由 + 证据检索 + 只读工具 + 规则计算 + 诊断组织 + 人工确认”。

## 3.1 讲案例时必须说清楚的中间结果

面试里不要只说“系统查了数据并给出诊断”，要把每一步的输入、处理、输出说出来：

| 环节 | 具体处理 | 具体结果样例 |
| --- | --- | --- |
| 会话创建 | 绑定用户、入口、campaign、时间窗、语言，生成 trace | `trace_id=tr_20250121_0001`，入口是 campaign sidebar |
| 权限校验 | RBAC 校验角色，ABAC 校验 account/advertiser/geo/tool scope | AM 可查询 `acct_1001/adv_2048/cmp_A`，不可导出 raw log |
| 意图识别 | 抽取 MMP、metric、event、时间窗和差异方向 | `intent=attribution_discrepancy_check`，`event=install`，`reported_gap=28%` |
| 场景拆解 | 归因智能体生成 7 个检查项 | 时区、归因窗口、事件映射、去重、postback、刷新延迟、campaign mapping |
| RAG 检索 | 中英文 query + entity query；权限过滤后 BM25/vector/entity exact 召回 | 选中 AppsFlyer install window、timezone SOP 等 3 条引用 |
| 工具查询 | 查平台 installs、MMP installs、postback 聚合状态 | 平台 1250，MMP 900，差异 28.0%，postback success rate 92.4% |
| 证据生成 | RAG citation 和工具结果转成 evidence object | `ev_metric_001` 支撑差异真实存在，`ev_policy_001` 支撑需对齐时区和窗口 |
| 诊断输出 | 场景智能体只基于 evidence object 生成结论 | 主假设是 timezone/window mismatch，postback 大面积失败被排除 |
| 人工确认 | 用户采纳、追问、升级或标记 Badcase | 无反馈不计入有效 AI 辅助排障会话 |

一句话表达：我会把这个案例讲成一条 trace，而不是讲成一次聊天。trace 里每一步都有输入、处理、输出和兜底。

# 4. 核心产品决策复盘

## 4.1 为什么用 RAG + 工具调用 + 规则引擎

| 方案 | 优点 | 问题 | 最终取舍 |
| --- | --- | --- | --- |
| 纯 LLM 问答 | 实现快，体验轻 | 容易幻觉，无法查实时数据，权限不可控 | 不适合直接用于投放归因诊断 |
| 纯规则/报表系统 | 结果稳定，可解释 | 难处理自然语言和复杂口径解释 | 适合作为指标计算与风控底座 |
| RAG 知识问答 | 能复用 SOP 和历史案例 | 无法证明当前账户是否真的异常 | 作为 context 和引用来源 |
| 工具调用 Agent | 能查实时数据并组织流程 | 工程复杂，需要权限和 Trace | 用于核心排障链路 |

最终方案：规则和 SQL 负责确定性计算，RAG 负责知识和口径，工具负责实时数据，LLM 负责意图识别、任务拆解、解释生成和证据组织。

## 4.2 为什么设计 evidence object

如果没有统一证据对象，模型输出很难被审计，也无法判断一句结论来自哪份文档或哪个工具结果。Evidence object 的价值是把不同来源统一成可追踪证据：

```text
source_type: metrics_query / attribution_report / postback_summary / knowledge_chunk / historical_case
source_id: 工具调用 ID 或文档 chunk ID
claim_supported: 这条证据支撑的判断
confidence: 系统置信度
visibility: internal / restricted
owner: 数据或知识负责人
retrieved_at: 获取时间
```

面试可讲：我不是让模型“相信上下文”，而是要求每个关键结论都能回到一个 evidence object。

## 4.3 为什么 Judge AI 不放在线上主链路

Judge AI 适合离线评测、灰度回归、人工抽检和 Badcase 复盘，不适合每次在线回答都强制经过。原因：

1. 会增加延迟和成本。
2. Judge 模型本身也可能判断错误。
3. 在线高风险更适合用规则实时拦截，比如越权、无证据强答、敏感信息。
4. Judge 更适合作为质量治理工具，而不是正常业务链路里的“裁判节点”。

# 5. 指标与价值怎么回答

## 5.1 北极星指标

推荐表达：

```text
我不会把模型调用次数或会话数作为北极星，因为这不能证明业务价值。
我定义的是“有效 AI 辅助投放归因排障会话数”：用户提出真实投放/归因问题，系统在权限范围内给出带证据的诊断，用户确认有效、采纳或完成后续动作。
这个指标能同时约束使用量和有效性。
```

## 5.2 如何证明节省时间

不能直接说“节省 30% 时间”是事实。更稳的回答是：

```text
我会先采集 2-4 周人工基线，比如同类投放异常和归因差异问题的 P50/P90 处理时长、升级率、返工率。
灰度后用同类场景对比 AI 辅助会话的处理时长。
计算公式是：有效会话数 × max(人工基线P50处理分钟 - AI辅助P50处理分钟, 0) / 60。
如果没有真实灰度数据，我只会把 30% 作为目标或假设，不会写成已经达成。
```

## 5.3 指标分层

| 层级 | 指标 | 面试解释 |
| --- | --- | --- |
| 业务 | 有效排障会话数、节省时间、升级率下降 | 证明项目有没有业务价值 |
| 体验 | 首版可用诊断时长、采纳率、追问次数 | 证明用户是否愿意用 |
| 质量 | 诊断命中率、核查完整率、引用准确率 | 证明 AI 是否可靠 |
| 安全 | 越权率、无证据强答率、敏感信息泄露率 | 证明能不能上线 |
| 成本 | 单会话成本、P95 延迟、强模型升级比例 | 证明能不能规模化 |

# 6. RAG 设计怎么回答

## 6.1 八段链路

1. 数据来源接入：SOP、指标口径、归因政策、历史工单、MMP 文档。
2. 文档解析清洗：保留标题、表格、代码块、链接，去掉隐私和模板噪音。
3. Chunk 切分：按业务主题和步骤切分，保留 overlap。
4. 元数据和权限标注：语言、区域、source type、敏感级别、Owner、生效时间。
5. 索引存储：Hybrid Search + Vector Search，实体和错误码走关键词召回。
6. Query 改写与召回：生成中文、英文、实体 query，先按权限过滤。
7. Rerank 与引用选择：按相关性、时效、同区域、reviewed 状态排序。
8. 反馈回流：引用无效、知识缺失、冲突口径进入 Badcase 和知识治理。


## 6.1.1 参数级 RAG 设计怎么讲

如果面试官继续追问“具体怎么做 RAG”，可以按下面讲：

1. 文档清洗不是统一切一刀。SOP 按排查步骤切，指标口径按指标定义和公式切，归因政策按 MMP/event/window 切，历史工单先脱敏再按完整案例摘要切。
2. Chunk 参数不是固定死值。SOP 用 300-600 tokens、overlap 80；指标口径用 200-450 tokens、overlap 60；历史案例 350-700 tokens、overlap 0，因为案例根因和解决动作不能被拆散。
3. 元数据必须服务权限和召回。核心字段包括 source_type、language、business_locale、sensitivity_level、reviewed_status、owner、effective_date、entity_tags、permission_scope。
4. Query 改写至少生成三类 query：英文语义 query、中文语义 query、实体过滤 query。实体 query 用于锁定 MMP、event、geo、source_type。
5. 召回采用混合检索：BM25 top 30 处理字段和实体精确命中，vector top 40 处理语义相似，entity exact top 20 保证同 MMP/同 event/同区域。
6. Rerank 不是只看模型分。最终分数由模型 rerank、vector、BM25、entity match、freshness、authority、locale match 和 penalty 组成。
7. 低于阈值不引用。MVP 中要求 `rerank_score >= 0.65` 且 `final_score >= 0.60`，否则只能进入待确认，不支撑主结论。
8. 召回结果不直接进 Prompt，而是转成 citation，再转成 evidence object，场景智能体只能基于 evidence object 生成结论。

可直接说的公式：

```text
final_score =
  0.45 × model_rerank_score
+ 0.15 × vector_score
+ 0.15 × bm25_score
+ 0.10 × entity_match_score
+ 0.05 × freshness_score
+ 0.05 × authority_score
+ 0.05 × locale_match_score
- penalty_score
```

## 6.2 RAG 常见追问

| 追问 | 回答要点 |
| --- | --- |
| 如何避免召回错文档 | 权限预过滤、source type 白名单、rerank、引用人工抽检、低相关度不强答 |
| 新旧文档冲突怎么办 | effective_date、reviewed_status、owner 参与排序；冲突时展示差异，不下确定结论 |
| 历史案例会不会误导 | 历史案例只做辅助证据，主结论必须有当前数据证据 |
| 中英混合怎么处理 | 保留英文实体，补充中文/英文 query，维护双语术语表 |
| 没有召回怎么办 | 不让模型自由编，转工具查询或输出待补充信息 |

# 7. 数据/API 依赖和兜底

面试里如果问“报表 API 不可用怎么办”，可以按降级顺序回答：

```text
实时 API > 有效缓存 > ETL 快照 > 人工上传报表 > 只输出待补充信息。
每一级都要显示数据新鲜度和置信度，不能让模型把旧数据当实时数据讲。
```

| 场景 | 处理 |
| --- | --- |
| 报表 API 超时 | 用最近缓存，标注数据时间，同时创建后台重试 |
| MMP API 未接入 | 支持 CSV/Excel 上传或粘贴关键字段，字段映射校验后再核对 |
| postback 明细不可查 | 只展示聚合摘要，不展示 raw URL、token、用户级日志 |
| ETL 延迟 | 使用上一批快照并显示批次号，关键结论进入人工确认 |
| 权限数据缺失 | 工具网关拒绝查询，不展示任何数据摘要 |

# 8. 模型选型和合规怎么讲

## 8.1 模型不是越强越好

这个项目的模型选型不是一个模型打天下，而是按任务拆分：

| 任务 | 需要的能力 | 适合模型 |
| --- | --- | --- |
| 总控路由 | JSON 稳定、低延迟、中英实体识别 | GPT-4o mini、Qwen2.5-72B、GLM-4-Plus |
| 投放诊断 | 基于证据解释，不自由计算 | GPT-4o mini + 规则引擎、Qwen2.5、DeepSeek-V3 |
| 复杂归因 | 多因素推理、冲突证据解释 | GPT-4o、DeepSeek-R1、Qwen QwQ |
| RAG 召回 | 中英混合语义检索 | text-embedding-3-large、bge-m3、Alibaba embedding |
| Rerank | 长文档引用准确率 | Cohere Rerank、bge-reranker、gte-rerank |

## 8.2 海外模型合规风险

稳妥表达：

```text
如果企业具备合规海外主体和数据处理协议，可以评估 OpenAI 等海外模型。
但广告数据和客户信息有合规风险，所以架构上必须模型可替换。
如果海外模型不可用，就切到 Qwen、DeepSeek、GLM、Kimi 或私有化部署方案。
关键是权限过滤、工具网关、证据对象和规则计算不依赖某个模型供应商。
```

# 9. 成本控制怎么讲

## 9.1 单次会话成本公式

```text
单次会话成本 =
  Σ(输入token × 输入单价 + 输出token × 输出单价)
+ Rerank成本
+ Embedding查询成本
+ 外部工具调用成本
```

所有 token 和工具调用都写入 Trace，按场景、模型、Prompt 版本统计。

## 9.2 降本策略

| 策略 | 说明 |
| --- | --- |
| 模型分层 | 路由和简单诊断用轻量模型，复杂归因再升级强模型 |
| 上下文压缩 | 工具结果转 evidence object，不把原始大表塞进 Prompt |
| 检索缓存 | 同权限、同 query、同知识版本可缓存 RAG 结果 |
| 工具缓存 | 同 campaign、同时间窗、同指标短期缓存 |
| Judge 离线化 | Judge AI 不进每次在线链路，降低延迟和成本 |
| Prompt 拆分 | 小任务小 Prompt，避免每一步都调用大模型 |

# 10. Corner Case 清单

| Corner Case | 风险 | 回答思路 |
| --- | --- | --- |
| 用户问题缺少 campaign 或时间窗 | 召回和取数都不准 | 先追问必要字段，不直接诊断 |
| 用户无账户权限 | 数据泄露 | 工具网关拦截，不返回摘要 |
| MMP 和平台时区不同 | 差异被误判为异常 | 核查清单强制覆盖 timezone |
| 归因窗口不同 | 安装/转化差异解释错误 | 同时展示平台和 MMP window |
| 事件映射不一致 | install/activation/purchase 口径混淆 | 通过 event mapping 表校验 |
| postback 延迟 | 当天数据低估 | 展示刷新延迟和 last_sync_time |
| 平台报表 API 超时 | 模型乱猜 | 使用缓存/快照或输出待补充 |
| RAG 召回旧文档 | 引用过期 SOP | reviewed_status 和 effective_date 降权/拦截 |
| 历史案例相似但不适用 | 误导结论 | 历史案例只做辅助，不做主证据 |
| 工具返回冲突 | 诊断不可信 | 展示冲突证据，转人工确认 |
| 用户要求写客户回复 | 超出当前范围 | 输出内部摘要，不生成可直接发送内容 |
| 用户诱导越权查询 | 合规风险 | 拒绝并提示权限范围 |
| 模型给出因果强结论 | 误导决策 | 强制区分 confirmed、likely、need_confirm |
| 成本突然升高 | 难以规模化 | 看模型升级比例、topK、重试、输出 token |

# 11. 面试高频问题答法

## 11.1 你在项目里的产品价值是什么

精简版：

```text
我的价值不是简单写一个 Prompt，而是把广告投放排障这个模糊场景拆成可落地的产品系统：定义 MVP 边界、用户旅程、证据对象、工具权限、RAG 知识治理、评测指标和灰度门禁。
```

展开要点：

1. 从高频业务问题中选出投放异常和归因差异两个 MVP 场景。
2. 把自然语言问题转成结构化意图、实体、权限和工具需求。
3. 设计 evidence object，约束 AI 结论必须可追溯。
4. 和研发/算法协作定义工具 schema、Prompt 变量、评测集和上线门禁。
5. 用业务指标而不是模型指标衡量价值。

## 11.2 为什么不是直接接 ChatGPT/大模型

```text
因为投放归因问题有实时数据、客户权限、业务口径和审计要求。
直接接大模型只能生成解释，无法证明当前账户发生了什么，也无法保证不会越权。
所以我把大模型放在意图识别、任务拆解和诊断组织位置，把事实来源交给 RAG、只读工具和规则引擎。
```

## 11.3 怎么评估这个 AI 产品是否可上线

```text
我会先看安全门禁，再看效果和效率。
安全上越权调用率、敏感信息泄露率、高风险自动执行率必须为 0。
效果上看意图识别、诊断命中、引用准确、归因核查完整率。
业务上看有效排障会话数、采纳率、升级率下降和节省时长。
灰度阶段还要看 Badcase 是否能闭环，否则指标好看也不能扩大。
```

## 11.4 如果算法说模型效果不稳定怎么办

```text
我会先拆问题来源：是意图错、检索错、数据错、推理错、还是输出格式错。
不同问题不能都归因给模型。
意图错补 query 和标签样本；检索错调 chunk、metadata、rerank；数据错修工具和口径；推理错补规则和 Prompt；输出错加强 schema 和校验。
```

## 11.5 如果研发说数据 API 接不齐怎么办

```text
我会把实时 API 依赖拆成优先级。
MVP 必须接核心报表和归因对比；postback 可以先做聚合摘要；MMP API 不齐时支持人工报表上传或 ETL 快照。
同时产品上明确展示数据新鲜度和置信度，不让用户误以为所有数据都是实时的。
```

# 12. 风险表达

面试中可以主动说这些风险，显得更真实：

1. 这个项目最难的不是模型调用，而是权限、证据和口径治理。
2. RAG 不是一次性建库，难点是文档 Owner、失效治理和 Badcase 回流。
3. 节省时间必须用人工基线和灰度数据证明，不能凭感觉说。
4. 客户可见话术要谨慎，第一阶段只做内部诊断和摘要，不直接对客户发送。
5. 如果数据 API、MMP 权限、历史工单脱敏拿不到，MVP 范围要收缩。
