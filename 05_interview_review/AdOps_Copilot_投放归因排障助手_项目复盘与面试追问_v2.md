# AdOps Copilot 投放归因排障助手项目复盘与面试追问 v2.0

本文档是面试准备文档，不是 PRD，也不是项目修改记录。

它的目标是从面试官可能追问的角度，帮助我讲清楚这个 AI 产品项目的业务背景、产品判断、AI 架构、落地细节、指标验证、成本控制和风险边界。

重要边界：

1. 当前材料是作品集和 PRD 方案，不能把未上线、未灰度、未验证的数据说成真实结果。
2. 涉及节省时间、采纳率、成本、ROI 时，应表达为“目标、估算方法、灰度验证方案”，不能说成已达成。
3. 面试表达要体现“我知道怎么落地”，而不是“我会调用大模型”。

# 1. 面试主叙事

## 1.1 30 秒版本

面试官问：你简单介绍一下这个项目。

可以这样答：

```text
这个项目是一个面向广告投放运营、AM 和技术支持的内部 AI 排障助手，主要解决投放效果异常和平台/MMP/客户后台归因数据不一致的问题。

我没有把它设计成开放式聊天机器人，而是设计成证据驱动的排障工作流：总控先做意图识别、权限校验和 workflow 路由，场景智能体按固定核查清单调用 RAG、报表、MMP 和 postback 工具，把结果转成 evidence object，再生成可追溯的内部诊断摘要。

项目重点不是单个模型能力，而是把 RAG、工具调用、权限、安全、评测、成本和 Badcase 回流做成可上线的 AI 产品闭环。
```

## 1.2 2 分钟版本

面试官问：这个项目为什么有价值？你具体做了什么？

可以这样答：

```text
广告投放和归因问题的特点是链路长、系统多、口径复杂。一线同事遇到客户反馈“转化下降”“MMP installs 比平台低”“postback 延迟”等问题时，往往要翻 SOP、查平台报表、查 MMP、看 postback 摘要、问技术支持，处理时长不稳定，新人也容易漏查。

所以我把项目定位为“投放归因排障助手”，优先解决两个高频场景：投放效果异常诊断和归因数据不一致核对，同时支持低风险的纯知识查询。SDK/API 深排、素材审核和客户回复生成先不放进本 PRD，避免第一阶段范围过大。

产品上，我定义了 MVP 范围、目标用户、端到端流程、结果卡、证据对象和灰度指标。AI 方案上，我把总控智能体、场景 workflow、RAG 知识检索、只读工具、规则引擎、Judge AI 和 Badcase 回流拆清楚。总控负责识别 intent、实体、风险信号和 workflow，固定排查步骤由 workflow 处理，模型负责自然语言理解、证据解释和摘要生成。

为了避免方案停留在文档层，我还设计了 prompt schema、risk/confidence 规则、端到端测试 case、成本测算和上线门禁。这个项目主要体现的是 AI 产品经理把业务问题拆成可验证、可评估、可控成本的 AI 系统的能力。
```

## 1.3 这段项目重点展示的能力

| 能力 | 项目里的体现 | 面试时要主动强调 |
| --- | --- | --- |
| 业务理解 | 理解广告投放、MMP、归因窗口、postback、指标口径 | 不是泛 AI 问答，而是 AdTech 场景排障 |
| 产品取舍 | 先做投放归因排障，后做 SDK/API、素材审核、客户回复 | MVP 边界清晰，不贪大 |
| AI 产品设计 | RAG + workflow + tool calling + evidence object | 不把大模型当黑盒 |
| 数据与指标 | 有效会话、节省时长、升级率、采纳率、成本 | 用业务指标而非模型调用数证明价值 |
| 风险控制 | 权限、越权、无证据强答、客户可见话术拦截 | 内部工具也要安全上线 |
| 工程落地意识 | prompt schema、工具 schema、Trace、灰度、回滚 | 能和研发/算法协作 |

# 2. 项目背景与切入点

## 2.1 为什么做这个项目

面试官可能问：

- 你为什么选择广告投放排障这个方向？
- 这个问题是真实痛点还是为了做 AI 而做 AI？
- 为什么不做一个通用知识库问答？

1 分钟回答：

```text
我选择投放归因排障，是因为这个场景同时满足高频、有数据证据、流程可沉淀和业务价值可衡量。

广告投放运营和 AM 经常遇到投放指标异常、平台和 MMP 数据不一致、postback 延迟、归因窗口口径不一致等问题。这类问题不是单纯查文档能解决，也不是单纯让大模型解释就可靠，它需要结合 SOP、指标口径、当前账户数据、MMP 报表和 postback 状态。

所以它很适合做成证据型 AI 产品：AI 负责理解问题、组织排查和生成摘要，事实来源来自 RAG、只读工具和规则计算。
```

展开要点：

| 判断维度 | 为什么适合做 AI 产品 |
| --- | --- |
| 高频 | 一线反复处理相似投放和归因问题，适合沉淀 workflow 和知识库 |
| 证据明确 | 可以查报表、MMP、postback、SOP，不是纯主观判断 |
| 价值可算 | 可以衡量处理时长、升级率、返工率、人均处理量 |
| 风险可控 | MVP 只做只读查询和内部摘要，不自动改预算、出价或配置 |
| 可扩展 | 底座可扩到 SDK/API 排障、素材审核、客户回复 |

不要这样说：

```text
我想做一个 AI Copilot 帮运营提效。
```

这句话太泛。要说清楚“哪个运营场景、什么问题、为什么 AI 适合、怎么衡量价值”。

## 2.2 为什么先做投放归因，而不是 SDK/API、素材审核、客户回复

面试官可能问：

- 为什么不一开始做全链路 AdOps Copilot？
- 客户回复不是更容易展示效果吗？
- SDK/API 排障是否更有技术含量？

推荐回答：

```text
我会先做投放归因排障，因为它的数据证据更结构化，风险更可控，也更容易建立评测集。

SDK/API 排障会涉及接入日志、设备环境、SDK 版本、回调链路和技术细节，数据权限和工具依赖更复杂。
素材审核涉及创意、政策和合规，评测标准和风险边界不同。
客户回复生成虽然体验直观，但涉及对外承诺、客户语气、赔偿、内部信息泄露，更适合放在已有内部诊断可信之后。

所以第一阶段先把内部只读排障底座跑通，后续再扩展到 SDK/API、素材审核和客户回复。
```

追问防守：

| 追问 | 回答方向 |
| --- | --- |
| 这是不是范围太小 | 小是为了 MVP 可验证；投放和归因本身已覆盖高频一线问题 |
| 为什么不做客户回复 | 对外内容风险更高，先输出内部诊断摘要，由 AM 人工确认 |
| 后续如何扩展 | 复用总控、权限、RAG、工具网关、证据对象和评测框架 |

## 2.3 目标用户是谁

面试官可能问：

- 谁会用这个产品？
- 不同用户的需求有什么差异？

回答结构：

| 用户 | 典型问题 | 他们关心什么 | 产品如何服务 |
| --- | --- | --- | --- |
| AdOps / 投放运营 | spend、install、CVR、CPA、ROI 异常 | 快速定位投放层面原因 | 指标拆解、异常归因、下一步排查 |
| AM / 客户成功 | 客户反馈平台和 MMP 数据不一致 | 快速拿到内部诊断口径 | 归因核对卡、证据引用、内部摘要 |
| 技术支持 | postback、事件映射、MMP 配置问题 | 判断是否需要技术介入 | 聚合状态、工具证据、升级上下文 |
| 产品/运营负责人 | 口径统一、知识复用、效率提升 | 看效果、风险、成本和 Badcase | 看板、评测、知识治理 |

推荐表达：

```text
我不会把所有用户都当成同一种“内部员工”。AdOps 更关心指标和操作建议，AM 更关心能否形成稳定客户沟通口径，技术支持更关心是否有足够证据判断要不要介入。因此结果页不能只有一段回答，而要有诊断结论、证据卡、已排除项、下一步动作和升级上下文。
```

# 3. MVP 边界与意图处理

## 3.1 MVP 包含什么

面试官可能问：

- 第一版具体做什么？
- 你怎么判断 MVP 范围？

推荐回答：

```text
MVP 包含三类意图：投放效果异常诊断、归因数据不一致核对、纯知识查询。

投放效果异常诊断解决 spend、click、install、CVR、CPA、ROI 等指标异常。
归因数据不一致核对解决平台、MMP、客户后台数据不一致。
纯知识查询解决 attribution window、postback delay、指标口径、MMP 概念等不需要查实时账户数据的问题。

MVP 不包含 SDK/API 深排、素材审核、客户回复生成、预算出价修改和合同赔偿，因为这些涉及不同工具链、权限和风控标准。
```

## 3.2 纯知识查询为什么也放进范围内

面试官可能问：

- 你前面说是排障助手，为什么又支持知识查询？
- 知识查询会不会让范围变大？

推荐回答：

```text
纯知识查询是范围内，但它不是排障 workflow，而是低风险 RAG workflow。

一线同事并不总是问“某个 campaign 为什么异常”，也会问“AppsFlyer attribution window 是什么”“postback delay 常见原因有哪些”“OEM install 口径怎么定义”。这些问题不需要查账户和 MMP 当前数据，如果硬进入排障 workflow，会增加工具调用、权限复杂度和成本。

所以我把它单独路由到 wf_knowledge_lookup_v1，只基于已审核知识库回答并展示引用。如果用户问题转成“查某个账户当前情况”，再重新进入排障 workflow。
```

追问防守：

| 追问 | 回答 |
| --- | --- |
| 知识查询会不会幻觉 | 只允许基于 reviewed 知识和引用回答；无可靠引用就说无法确认 |
| 会不会泄露客户数据 | 知识查询不调用账户、MMP、postback 工具 |
| 怎么控制成本 | 轻量模型 + RAG 缓存 + 不走复杂诊断 |

## 3.3 范围外意图怎么兜底

面试官可能问：

- 用户问范围外问题怎么办？
- 大模型能回答，为什么不顺手回答？

推荐回答：

```text
我不会把范围外问题直接丢给大模型自由回答。

总控会先把意图分成支持范围、纯知识查询、范围外和高风险四类。
SDK/API 深排、素材审核、客户回复生成、预算/出价修改、合同赔偿都不进入当前 workflow。

兜底方式分三种：如果只是概念性知识，可以转成知识查询；如果需要后续 PRD 能力，提示当前阶段不支持并建议转人工或创建工单；如果涉及越权、敏感数据或操作变更，则拒绝并记录风险。
```

例子：

| 用户问题 | 路由 | 输出 |
| --- | --- | --- |
| `What is postback delay?` | `wf_knowledge_lookup_v1` | 引用知识库解释 |
| `Why did SDK callback fail on device X?` | `wf_out_of_scope_fallback_v1` | 提示 SDK/API 深排不在当前阶段，可提供需要的信息清单 |
| `Help me reply to the client.` | `wf_out_of_scope_fallback_v1` | 可生成内部诊断摘要，不生成客户可直接发送文本 |
| `Change bid to recover volume.` | `wf_refusal_v1` 或人工确认 | 不执行操作变更 |
| `Show raw logs for all advertisers.` | `wf_refusal_v1` | 拒绝越权和敏感数据请求 |

# 4. 端到端案例讲法

## 4.1 请你举一个完整案例

面试官可能问：

- 你能不能讲一个用户从提问到拿到结果的完整流程？
- 不要只讲架构，讲一个真实使用场景。

推荐案例：

```text
假设 AM 收到客户反馈：昨天 campaign A 在平台显示 1250 installs，但 AppsFlyer 只有 900，差异约 28%，客户怀疑平台数据不准。
```

1 分钟回答：

```text
用户在 campaign 页面打开 Copilot，输入“AppsFlyer installs are 28% lower than platform for campaign A yesterday”。

系统先创建 trace，绑定用户、账户、campaign、时间窗和语言。总控识别这是归因数据不一致核对，校验用户是否有该 account、campaign 和 MMP 查询权限，再选择 wf_attribution_discrepancy_v1。

workflow 加载固定核查清单，包括时区、归因窗口、事件映射、去重、postback、MMP 刷新延迟和渠道映射。场景智能体按清单调用 RAG、平台报表、MMP 报表和 postback 摘要工具。

工具返回后，系统把数据和引用转成 evidence object。诊断结果会说明差异确实存在，postback success rate 正常，因此优先检查时区和归因窗口，其次检查 MMP 刷新延迟。输出里会展示证据、已排除项、下一步动作和需要人工确认的地方。
```

## 4.2 讲案例时要说出的中间产物

面试官真正想听的是“你有没有落地感”，所以要讲中间产物。

| 环节 | 中间产物 | 为什么重要 |
| --- | --- | --- |
| 会话创建 | `trace_id`、用户角色、入口、语言 | 后续可追踪 |
| 权限校验 | account/campaign/MMP/postback scope | 防止越权 |
| 总控路由 | intent、entities、risk_signals、selected_workflow | 决定走哪个 workflow |
| 固定核查清单 | timezone、window、event mapping、dedupe、postback | 避免漏查 |
| RAG 结果 | citation、source、reviewed_status、effective_date | 保证口径可信 |
| 工具结果 | platform installs、MMP installs、postback success rate | 支撑事实判断 |
| Evidence object | evidence_id、claim_supported、visibility、confidence | 让结论可追溯 |
| 诊断卡 | 可能原因、已排除项、下一步、置信度 | 用户可执行 |
| 反馈 | 采纳、继续追问、升级工单、Badcase | 闭环优化 |

## 4.3 如果用户信息不完整怎么办

面试官可能问：

- 用户只说“昨天数据不对”，你怎么处理？
- 会不会让模型自己猜 campaign？

推荐回答：

```text
不会让模型猜。总控会按 intent 复算必填字段。

归因核对至少需要 account/campaign 或 app、MMP、event、time range。如果缺关键字段，selected_workflow 不能进入诊断，而是进入 clarification，向用户追问缺失信息。

这里 missing_fields 和 next_action 不能只采用模型自报，要由 workflow 按必填实体复算。
```

# 5. AI 架构与核心设计

## 5.1 为什么不是直接接 ChatGPT

面试官可能问：

- 为什么不直接用 ChatGPT 或企业知识库问答？
- 你这个方案是不是过度设计？

推荐回答：

```text
直接接大模型可以生成解释，但不能证明当前账户发生了什么，也无法保证权限、口径和证据可追溯。

投放归因问题涉及实时或准实时数据、客户权限、MMP 口径、postback 状态和业务规则。大模型不能自由计算指标，也不能绕过权限查数据。

所以我把大模型放在适合的位置：理解自然语言、抽取实体、解释证据、生成摘要。确定性部分交给 workflow、规则引擎、SQL、工具网关和权限系统。
```

简短金句：

```text
这个项目不是让 LLM 替代排障流程，而是让 LLM 驱动一个受控的排障流程。
```

## 5.2 总控智能体具体做什么

面试官可能问：

- 总控 agent 的输入输出是什么？
- 它是不是每次都让模型决定下一步？

推荐回答：

```text
总控不是一个自由决策的 Agent，而是 LLM + 规则状态机。

它的输入包括用户问题、会话上下文、用户权限摘要、当前入口、可用工具清单和产品范围。
模型先输出候选 intent、entities、risk_signals、confidence_components、missing_fields、candidate_tools 和 selected_workflow。

然后 Schema Guard 校验 JSON，规则层复算 risk_level_final、confidence_final、missing_fields_final、next_action_final 和 selected_workflow_final。
最终只有通过规则校验的 workflow 才能执行。
```

总控 workflow：

| 步骤 | 处理 | 模型能做什么 | 规则系统做什么 |
| --- | --- | --- | --- |
| 创建上下文 | 生成 trace，绑定用户、入口、语言 | 无 | 后端确定性处理 |
| 权限预检 | 读取角色、账户、客户、工具权限 | 无 | RBAC/ABAC |
| 意图识别 | 判断投放诊断、归因核对、知识查询、范围外 | 输出候选 intent | 校验枚举和范围 |
| 实体抽取 | account、campaign、MMP、event、time range | 抽取候选实体 | 校验必填字段 |
| 风险信号 | 是否涉及越权、客户回复、操作变更 | 输出 risk_signals | 复算 risk_level |
| workflow 分发 | 候选 workflow | 输出建议 | 规则层最终选择 |
| 安全交付 | 检查无证据、越权、敏感信息 | 可辅助改写 | 规则拦截 |

## 5.3 workflow 和 LLM 怎么分工

面试官可能问：

- 既然任务拆解固定，为什么还要 Agent？
- 为什么不用纯 workflow？

推荐回答：

```text
稳定步骤用 workflow，不稳定理解和解释用 LLM。

比如归因核对的时区、归因窗口、事件映射、postback、去重、渠道映射是固定必查项，不应该让模型每次自由拆。
但用户问题是自然语言，可能中英混合、缺字段、多意图、带业务上下文，这部分适合用 LLM 做理解和实体抽取。

场景智能体的价值不是创造检查清单，而是基于固定清单判断需要哪些证据、解释工具结果、组织原因排序和输出内部诊断摘要。
```

分工表：

| 任务 | 负责人 |
| --- | --- |
| 权限、必填字段、风险等级、工具执行顺序 | workflow / 规则系统 |
| 指标差异、比例计算、阈值判断 | SQL / 规则引擎 |
| 自然语言理解、实体抽取、证据解释 | LLM |
| 口径知识、SOP、历史案例 | RAG |
| 实时或准实时事实 | 只读工具 |
| 高风险输出 | 规则拦截 + 人工确认 |

## 5.4 为什么设计 Evidence Object

面试官可能问：

- 你怎么防止模型幻觉？
- 诊断结论怎么审计？

推荐回答：

```text
我要求关键结论必须绑定 evidence object。

RAG 引用、平台报表、MMP 报表、postback 摘要和历史案例都先转成统一证据对象，包含 source_type、source_id、claim_supported、visibility、retrieved_at、confidence 和 owner。

场景智能体生成诊断时不能直接引用未结构化上下文，而是基于 evidence object 输出结论。这样每一句关键判断都能追溯到数据或文档来源。
```

Evidence object 示例字段：

```json
{
  "evidence_id": "ev_metric_001",
  "source_type": "metrics_query",
  "source_id": "tool_call_20250121_001",
  "claim_supported": "platform_installs=1250 and mmp_installs=900",
  "visibility": "internal",
  "retrieved_at": "2025-01-21T10:00:00Z",
  "confidence": 0.94
}
```

# 6. RAG 设计追问

## 6.1 RAG 链路怎么设计

面试官可能问：

- 你的 RAG 具体怎么做？
- 怎么避免只写“向量库检索”这种泛泛方案？

推荐回答：

```text
我把 RAG 拆成 8 段：数据来源、文档清洗、chunk 切分、元数据和权限标注、索引存储、query 改写、混合召回和 rerank、引用反馈回流。

这个项目的 RAG 不是直接回答，而是给场景智能体提供可引用的口径、SOP 和历史案例。RAG 输出要先经过权限过滤和 source type 白名单，再转成 evidence object。
```

8 段链路：

1. 数据来源：投放 SOP、指标口径、归因政策、MMP 文档、脱敏历史案例。
2. 文档清洗：保留标题、表格、公式、步骤、链接，去掉隐私和重复模板。
3. Chunk 切分：SOP 按步骤，指标按定义和公式，历史案例按完整案例摘要。
4. 元数据标注：language、source_type、reviewed_status、effective_date、owner、permission_scope。
5. 索引存储：BM25 + vector + metadata filter。
6. Query 改写：英文语义 query、中文语义 query、实体 query。
7. Rerank：相关性、实体匹配、时效、权威、区域、权限综合排序。
8. 反馈回流：无引用、引用错误、口径冲突进入 Badcase 和知识治理。

## 6.2 Chunk 和 metadata 怎么设计

面试官可能问：

- Chunk 多大？
- 元数据具体有什么？

推荐回答：

| 文档类型 | Chunk 策略 | 关键 metadata |
| --- | --- | --- |
| SOP | 按排查步骤切，300-600 tokens，overlap 80 | step_name、scenario、reviewed_status、owner |
| 指标口径 | 一个指标定义或强相关指标一组，200-450 tokens，overlap 60 | metric_name、formula_version、effective_date |
| 归因政策 | 按 MMP、event、window、geo 切 | mmp、event_type、geo、policy_version |
| 历史案例 | 脱敏后按完整案例摘要切，350-700 tokens | issue_type、root_cause、resolution、confidence |

核心解释：

```text
广告场景有大量英文缩写和实体，比如 CPI、CPA、ROAS、postback、SKAN、AppsFlyer，所以不能只靠语义相似。metadata 必须参与召回和过滤，尤其是 MMP、event、geo、source_type、reviewed_status 和 permission_scope。
```

## 6.3 如何避免召回错误文档

面试官可能问：

- RAG 召回错了怎么办？
- 新旧口径冲突怎么办？

推荐回答：

```text
我会在召回前、召回中、生成前做三层控制。

召回前先按用户权限和 source_type 过滤，避免拿到不该看的文档。
召回中用 BM25、vector、entity exact 和 rerank 混合排序，防止只靠向量相似召回错场景。
生成前要求引用达到分数阈值，并检查 reviewed_status、effective_date 和 owner。低于阈值的 chunk 不能支撑主结论。
```

冲突处理：

| 情况 | 处理 |
| --- | --- |
| 新旧文档冲突 | reviewed + effective_date 优先；冲突时展示差异并提示待确认 |
| 历史案例相似但场景不同 | 只作为辅助，不作为主证据 |
| 没有高质量引用 | 不回答确定结论，提示知识库暂无可靠口径 |
| 权限过滤后无结果 | 不能绕过权限，也不能用其他客户案例补充 |

# 7. 权限、安全和风险控制

## 7.1 权限模型怎么设计

面试官可能问：

- 内部系统是不是就不用管权限？
- 怎么防止 AM 看到不该看的客户数据？

推荐回答：

```text
内部工具也必须做权限控制，尤其广告数据和客户数据涉及账户、区域、客户和敏感字段。

我会用 RBAC + ABAC。RBAC 控制角色能力，比如 AdOps、AM、技术支持能用哪些功能；ABAC 控制 account、advertiser、geo、data sensitivity 和 tool scope。

工具调用前做权限校验，RAG 检索前也做权限过滤，输出前再检查 evidence visibility。模型本身不拥有绕过权限的能力。
```

## 7.2 risk_level 怎么判断

面试官可能问：

- risk_level 是模型判断的吗？
- 你怎么保证它靠谱？

推荐回答：

```text
risk_level 不能直接相信模型自报。

模型只输出 risk_signals，例如 account_data_read、mmp_data_read、postback_summary_read、customer_visible_reply、permission_gap、sensitive_raw_data、operational_change。

最终 risk_level 由规则层复算：只要有权限缺口、敏感原始数据、操作变更、客户可见回复，就判 high；只读账户/MMP/postback 诊断一般是 medium；纯知识查询且不涉及账户数据才是 low。
```

容易被追问的边界：

```text
用户说“客户反馈数据不一致”，不等于请求生成客户回复。
只有用户明确要求“帮我回复客户/发给客户/承诺恢复时间/解释赔偿”，才打 customer_visible_reply 并进入高风险或范围外兜底。
```

## 7.3 confidence 怎么计算

面试官可能问：

- 置信度是怎么来的？
- 模型说 0.9 你就展示 0.9 吗？

推荐回答：

```text
不会。confidence 也不采用模型主观自评，而是组件化复算。

总控阶段看 intent_match、entity_completeness、permission_fit、phase_fit 和 tool_coverage。
诊断阶段还要看是否有当前报表、MMP 数据、postback 摘要、知识引用、必查项覆盖和证据冲突。

如果只有知识引用没有当前数据，置信度不能超过 0.40；缺平台或 MMP 任一核心报表，不能超过 0.45；工具降级到缓存或人工上传，要下调并展示数据新鲜度。
```

公式：

```text
confidence_final =
  0.35 × intent_match_score
+ 0.25 × entity_completeness_score
+ 0.20 × permission_fit_score
+ 0.10 × phase_fit_score
+ 0.10 × tool_coverage_score
```

## 7.4 客户可见输出怎么处理

面试官可能问：

- AM 最想要的是回复客户，你为什么不做？
- 如果用户让 AI 写客户回复怎么办？

推荐回答：

```text
第一阶段我不生成客户可直接发送的回复，只生成内部诊断摘要。

原因是客户回复涉及对外承诺、赔偿、内部成本、其他客户信息和法律合规风险。当前 PRD 的目标是先把内部诊断做准，客户沟通由 AM 人工确认。

如果用户要求“帮我回复客户”，系统会提示当前阶段只支持内部诊断摘要，并可输出“可供 AM 参考的内部要点”，但不会生成可直接发送文本。
```

# 8. 数据工具与兜底

## 8.1 工具怎么设计

面试官可能问：

- 你有哪些工具？
- 工具调用怎么防止模型乱调？

推荐回答：

```text
工具都是只读工具，并且通过 Tool Gateway 调用。模型不能自由拼 SQL 或任意 API，只能填 schema 允许的参数。

核心工具包括 campaign metrics、account status、MMP attribution report、postback summary、knowledge search 和 similar cases。每个工具都有输入字段、输出字段、权限要求、超时、重试和证据对象转换规则。
```

工具示例：

| 工具 | 用途 | 权限 |
| --- | --- | --- |
| `get_campaign_metrics` | 查 spend、click、install、CVR、CPA、ROI | account/campaign scope |
| `get_mmp_attribution_report` | 查 MMP 侧 install/event 数据 | MMP access |
| `get_postback_summary` | 查聚合 success/delay/reject | postback summary permission |
| `search_knowledge_base` | 查 SOP、口径、FAQ | knowledge_read |
| `search_similar_cases` | 查脱敏历史案例 | case_read + sensitivity filter |

## 8.2 数据 API 不可用怎么办

面试官可能问：

- 如果 MMP API 没接上，这个项目是不是就不能做？
- 工具超时怎么办？

推荐回答：

```text
我会把数据依赖设计成分级兜底，而不是 API 不可用就让模型猜。

优先级是实时 API、有效缓存、ETL 快照、人工上传报表、只输出待补充信息。每一级都要展示数据新鲜度和置信度。
```

兜底表：

| 问题 | 兜底 | 输出限制 |
| --- | --- | --- |
| 平台报表 API 超时 | 使用短期缓存或 ETL 快照 | 标注数据时间 |
| MMP API 未接入 | 支持 CSV/Excel 上传关键字段 | 只做表内核对 |
| postback 明细不可查 | 返回聚合摘要 | 不展示 raw URL/token |
| 权限数据缺失 | 工具网关拒绝 | 不返回任何客户数据 |
| 所有数据都不可用 | 输出待补充清单 | 不做结论 |

# 9. 模型选型与成本

## 9.1 模型怎么选

面试官可能问：

- 为什么选国内模型？
- 海外业务英文系统，用国内模型靠谱吗？
- 为什么不用一个最强模型全搞定？

推荐回答：

```text
模型选型不是越强越好，而是任务匹配、成本、合规和可替换性共同决定。

这个项目的在线链路优先考虑 Qwen、DeepSeek、GLM、Kimi 或私有化部署方案，因为企业主体、数据边界和成本更可控。OpenAI、Gemini、Cohere 等海外模型可以作为合规可用时的对照评测或灰度兜底，但不能让核心架构依赖它们。

路由任务重视 JSON 稳定和低成本，诊断任务重视证据解释，复杂归因才需要推理模型。Embedding 和 rerank 则看中英文混合召回、专有名词和引用准确率。
```

任务匹配：

| 任务 | 首选方向 | 关键评测 |
| --- | --- | --- |
| 总控路由 | Qwen / DeepSeek / GLM 轻量或通用模型 | intent、JSON、实体、workflow 分流 |
| 投放诊断 | DeepSeek-V3 + 规则引擎 | 证据解释、原因排序 |
| 复杂归因 | DeepSeek-R1 / Qwen QwQ | 多步推理、冲突证据 |
| Embedding | bge-m3 / Alibaba embedding | 中英混合召回 |
| Rerank | bge-reranker / Alibaba gte-rerank | 引用准确率 |

## 9.2 国内模型英文能力如何验证

面试官可能问：

- 海外业务英文系统，中文模型能不能处理？

推荐回答：

```text
我不会口头争论“能不能”，而是把它变成评测集问题。

评测集要覆盖英文 query、中文 query、中英混合 query、广告指标缩写、MMP 名词和真实业务表达。比如 installs、postback delay、attribution window、SKAN、SAN、ROAS、campaign mapping。

如果国内模型在复杂英文场景上不稳，可以用更强模型做灰度升级或离线抽检，而不是默认全链路用最贵模型。
```

## 9.3 成本大概是多少

面试官可能问：

- 这个系统跑起来贵不贵？
- 国内模型和海外模型成本差多少？
- 你的 ROI 怎么算？

推荐回答：

```text
我会先按会话类型估算，而不是只说“成本可控”。

假设 35% 是纯知识查询、55% 是标准投放/归因排障、10% 是复杂归因排障。国内模型方案按 DeepSeek-chat/reasoner + 国内 embedding/rerank 粗算，平均大约 0.041 元人民币/会话，1 万次会话约 412 元，10 万次约 4122 元。

海外模型方案按 GPT-4o mini、GPT-4o、OpenAI embedding 和 Cohere/Pinecone rerank 粗算，平均大约 0.043 美元/会话，折合约 0.31 元人民币/会话，1 万次约 430 美元。

这个测算的意义不是报价，而是说明成本主要来自标准/复杂排障中的强模型长上下文诊断，所以产品策略上要做模型分层、RAG 缓存、工具结果压缩和复杂任务升级控制。
```

成本表：

| 场景 | 流量占比假设 | 国内模型单次成本 | 海外模型单次成本 |
| --- | ---: | ---: | ---: |
| 纯知识查询 | 35% | 约 ¥0.028 | 约 $0.0035，折合约 ¥0.025 |
| 标准投放/归因排障 | 55% | 约 ¥0.041 | 约 $0.0557，折合约 ¥0.40 |
| 复杂归因排障 | 10% | 约 ¥0.086 | 约 $0.1118，折合约 ¥0.81 |
| 加权平均 | 100% | 约 ¥0.041 / 会话 | 约 $0.043 / 会话，折合约 ¥0.31 / 会话 |

成本埋点：

| 字段 | 用途 |
| --- | --- |
| `workflow_type` | 按场景拆成本 |
| `model_name` / `provider` | 看模型分层是否有效 |
| `input_tokens` / `output_tokens` | 计算大模型费用 |
| `cached_input_tokens` | 计算缓存收益 |
| `embedding_tokens` | 计算 RAG 查询成本 |
| `rerank_requests` | 监控 rerank 成本 |
| `tool_retry_count` | 发现工具重试导致的成本异常 |
| `price_config_version` | 支持历史成本复算 |

# 10. 指标、价值与 ROI

## 10.1 北极星指标是什么

面试官可能问：

- 你怎么衡量这个产品成功？
- 为什么不用调用量或 DAU？

推荐回答：

```text
我不会把模型调用次数或聊天次数作为北极星，因为它们不能证明业务价值。

我定义的北极星是“有效 AI 辅助投放归因排障会话数”。它必须同时满足：用户提出真实投放或归因问题，用户有权限，Copilot 给出带证据的诊断或核查结果，用户确认有效、采纳、复制内部摘要、继续按建议查询或升级工单。
```

排除项：

| 不计入 | 原因 |
| --- | --- |
| 测试账号 | 不代表真实业务 |
| 权限不足 | 没有交付诊断 |
| 范围外问题 | 不是当前 MVP 价值 |
| 纯闲聊 | 不产生排障价值 |
| 工具失败未交付 | 用户没有获得有效帮助 |

## 10.2 如何证明节省时间

面试官可能问：

- 你怎么证明节省了 30% 时间？
- 没上线怎么讲价值？

推荐回答：

```text
如果没有灰度数据，我不会说已经节省 30%。我会说这是目标或估算，并说明验证方法。

上线前先采集 2-4 周人工基线，按问题类型统计人工 P50/P90 处理时长、升级率和返工率。灰度后用 Copilot trace 和工单数据统计 AI 辅助 P50。

节省时间 = 有效会话数 × max(人工基线 P50 - AI 辅助 P50, 0) / 60。
```

示例估算只用于说明方法：

| 问题类型 | 月有效会话假设 | 人工基线 P50 | AI 辅助 P50 | 月节省小时 |
| --- | ---: | ---: | ---: | ---: |
| 归因数据不一致 | 80 | 45 分钟 | 28 分钟 | 22.7 |
| 投放效果异常 | 60 | 40 分钟 | 24 分钟 | 16.0 |
| 归因/指标口径查询 | 40 | 12 分钟 | 5 分钟 | 4.7 |
| 合计 | 180 | - | - | 43.4 |

关键表达：

```text
这些不是已达成结果，而是灰度前的测算模型。真实结果要用埋点、工单系统和人工抽检复算。
```

## 10.3 质量指标怎么设计

面试官可能问：

- 除了业务指标，AI 产品质量怎么看？

指标分层：

| 层级 | 指标 | 说明 |
| --- | --- | --- |
| 业务 | 有效会话数、节省时长、升级率下降、返工率下降 | 是否真的提效 |
| 体验 | 首版可用诊断时长、采纳率、追问次数、满意度 | 用户是否愿意用 |
| 效果 | intent 准确率、诊断命中率、核查完整率、引用准确率 | AI 是否可靠 |
| 安全 | 越权率、敏感信息泄露率、无证据强答率 | 能否上线 |
| 成本 | 单会话成本、P95 延迟、强模型升级比例 | 能否规模化 |

# 11. Prompt、评测和上线

## 11.1 Prompt 怎么设计

面试官可能问：

- 你只是写了一段 Prompt 吗？
- Prompt 里的变量和输出怎么控制？

推荐回答：

```text
我会把 Prompt 当成可版本化的产品配置，不是一次性文案。

Prompt 分为系统约束、任务说明、业务上下文、工具结果、证据对象、禁止项和输出 schema。所有变量都明确来源，比如 routing_result、permission_scope、evidence_objects、business_rules、tool_results。

输出必须是可解析 JSON，关键结论必须绑定 evidence_id，缺少证据时只能追问或输出待确认，不能自由补齐。
```

## 11.2 为什么做端到端 Prompt 测试

面试官可能问：

- Prompt 效果怎么验证？
- 怎么证明它不是纸面方案？

推荐回答：

```text
因为 prompt 文档写得完整，不代表模型会稳定遵守。

我会用脱敏模拟 case 做端到端回归：固定用户问题、权限、工具返回值和预期结果，让模型跑总控路由和诊断输出，再用程序断言 JSON schema、intent、risk、confidence、selected_workflow、工具约束、必查项覆盖和 evidence_id。
```

当前已有验证思路：

| 测试项 | 覆盖 |
| --- | --- |
| 纯知识查询 | 应进入 `wf_knowledge_lookup_v1` |
| 归因差异 | 应进入 `wf_attribution_discrepancy_v1` |
| 越权 raw log | 应拦截为 high risk |
| 缺 campaign/account | 应追问，不进入诊断 |
| 诊断 case | 检查时区、窗口、postback、证据引用 |

## 11.3 Judge AI 放在哪里

面试官可能问：

- 为什么不让 Judge AI 每次在线判断？
- Judge AI 有什么价值？

推荐回答：

```text
Judge AI 不作为每次在线回答的必经节点，而用于离线评测、灰度回归、人工抽检和 Badcase 复盘。

原因是在线强制 Judge 会增加延迟和成本，而且 Judge 模型本身也可能错。在线安全更适合用规则实时拦截，比如越权、敏感信息、无证据强答和客户可见话术。

Judge AI 的价值是做版本对比和质量治理：检查证据完整性、引用准确性、核查清单覆盖、安全风险和可执行性。
```

## 11.4 灰度上线门禁

面试官可能问：

- 什么条件你才敢上线？

推荐回答：

```text
我会先看安全，再看效果，再看效率。

越权调用、敏感信息泄露、高风险自动执行必须为 0。然后看核心黄金集里的 intent、引用准确、诊断命中和核查完整率是否达标。灰度阶段还要看 Badcase 是否能闭环，否则即使短期指标好也不能扩大。
```

上线门禁示例：

| 类型 | 门槛 |
| --- | --- |
| 安全 | 越权调用率 0，敏感信息泄露率 0 |
| 效果 | 核心 intent 准确率、引用准确率达到灰度门槛 |
| 体验 | P95 延迟可接受，复杂工具链可流式展示进度 |
| 成本 | 平均会话成本和强模型升级比例可监控 |
| 运营 | Badcase 可标记、分派、修复、回归 |

# 12. 协作与 PM 角色

## 12.1 你作为产品经理具体负责什么

面试官可能问：

- 你在这个项目里的贡献是什么？
- 研发和算法做了很多，你的产品价值在哪里？

推荐回答：

```text
我的价值不是替代研发或算法，而是把业务问题拆成可落地的产品系统。

我负责定义 MVP 边界、用户旅程、核心场景、结果卡、证据对象、工具 schema、Prompt 变量、评测指标、灰度门禁和风险边界。和算法协作时，我提供黄金集、业务标签和错误分类；和研发协作时，我明确工具输入输出、权限逻辑、Trace 字段和上线监控；和业务协作时，我确认 SOP、指标口径和人工基线。
```

职责表：

| 模块 | PM 负责 | 研发/算法协作 |
| --- | --- | --- |
| 场景范围 | 定义 MVP 和范围外 | 评估实现成本 |
| Prompt/schema | 定义变量、禁止项、输出字段 | 实现结构化输出和版本管理 |
| RAG | 定义知识来源、元数据、审核流程 | 建索引、召回、rerank |
| 工具 | 定义用途、权限、证据字段 | 实现 API 和 Tool Gateway |
| 评测 | 定义黄金集、指标、Badcase 分类 | 跑模型评测和回归 |
| 灰度 | 定义门禁和看板 | 实现发布、监控、回滚 |

## 12.2 如果研发说工期太长怎么办

面试官可能问：

- 这么多模块怎么落地？
- 第一版会不会太重？

推荐回答：

```text
我会把 MVP 拆成三层。

第一层是知识查询和总控底座，先打通会话、权限、Trace、RAG 和基础路由。
第二层接入核心报表和 MMP 对账工具，覆盖投放异常和归因差异两个高频场景。
第三层再补评测后台、Badcase 回流和更多工具。

如果研发资源有限，我会优先保证只读工具、权限、证据对象和核心 workflow，而不是先做复杂 UI 或客户回复。
```

# 13. 常见压力追问

## 13.1 如果面试官说“这不就是 RAG 套壳”怎么办

回答：

```text
如果只是查 SOP，我同意它会变成 RAG 套壳。但这个项目的关键不是知识问答，而是投放归因排障。

它需要把用户问题路由到固定 workflow，调用当前账户报表、MMP 数据和 postback 摘要，把结果转成 evidence object，再用规则约束风险、置信度和输出。RAG 只是 context 来源之一，不是最终答案来源。
```

## 13.2 如果面试官说“为什么要 Agent，报表看板不就行了”怎么办

回答：

```text
报表看板能展示数据，但不能自动把用户问题转成排查路径，也不能把 SOP、口径、MMP 数据和 postback 状态组织成可执行诊断。

这个产品不是替代报表，而是在报表之上做自然语言入口、证据组织、原因排序和下一步动作建议。
```

## 13.3 如果面试官说“模型错了怎么办”怎么办

回答：

```text
我会先拆错在哪里：意图错、检索错、工具数据错、规则错、推理错还是表达错。

意图错补标注和 few-shot；检索错调 chunk、metadata 和 rerank；工具错查 API 和口径；规则错修 workflow；推理错补 evidence 约束和评测样本；表达错调输出 schema。

不能把所有 badcase 都归因于模型，也不能只靠换模型解决。
```

## 13.4 如果面试官问“怎么处理多轮对话”怎么办

回答：

```text
多轮对话要绑定 session state 和 trace。

每轮都记录已确认实体、已调用工具、已有 evidence、用户新补充信息和当前 workflow 状态。用户补充 campaign 或时间窗后，可以从 clarification 切回诊断 workflow；如果用户从知识查询转成账户问题，要重新做权限校验。

不能简单把完整历史塞进 prompt，否则成本高、上下文污染，也难审计。
```

## 13.5 如果面试官问“你怎么避免过度包装”怎么办

回答：

```text
我会明确区分已完成的方案设计、可运行的 prompt 测试、灰度目标和待验证数据。

比如成本、节省时间、采纳率这些，我可以讲估算公式和埋点方案，但不会说已经上线达成。项目价值在于我能把 AI 产品落地逻辑拆清楚，而不是虚构业务结果。
```

# 14. Corner Case 快速清单

| 场景 | 风险 | 产品处理 |
| --- | --- | --- |
| 缺 campaign / account / time range | 模型猜实体 | 追问必要字段 |
| 用户无权限 | 泄露数据 | 工具网关拦截 |
| 平台和 MMP 时区不同 | 误判异常 | 固定核查 timezone |
| 归因窗口不同 | 口径混淆 | 展示双方 window |
| event mapping 不一致 | install/activation 混淆 | 校验事件映射表 |
| postback 延迟 | 当天数据低估 | 展示 last_sync_time 和 delay |
| RAG 召回旧文档 | 口径过期 | effective_date 和 reviewed_status |
| 历史案例误导 | 以偏概全 | 历史案例只做辅助 |
| 工具返回冲突 | 强行下结论 | 标记冲突并转人工确认 |
| 用户要求客户回复 | 对外风险 | 输出内部摘要，不生成可直接发送文本 |
| 用户要求改预算/出价 | 操作风险 | 拒绝或人工审批 |
| 成本突然升高 | 难规模化 | 看强模型升级比例、topK、重试、输出 token |

# 15. 面试前自检

## 15.1 这个项目不能乱说的内容

| 不建议说 | 更稳的说法 |
| --- | --- |
| 已经节省 30% 时间 | 目标是降低处理时长，灰度后用人工基线和 Copilot trace 验证 |
| 已上线大规模使用 | 当前是 PRD/作品集方案，可讲可运行 prompt 验证和落地设计 |

| 用 GPT-4o 就能解决 | 模型只是组件，关键是权限、证据、工具和 workflow |
| AI 自动回复客户 | 第一阶段只做内部诊断摘要，客户回复需人工确认 |

## 15.2 面试时必须主动讲出的 5 个点

1. 为什么选投放归因排障：高频、有证据、可衡量、风险可控。
2. 为什么不是纯 LLM：实时数据、权限、口径和审计要求。
3. workflow 与 LLM 分工：稳定排查用 workflow，自然语言理解和证据解释用 LLM。
4. 如何防幻觉：RAG + 工具 + evidence object + 规则复算 risk/confidence。
5. 如何证明价值：有效会话、人工基线、节省时长、升级率、成本看板。

## 15.3 如果只剩 1 分钟怎么收尾

```text
我认为这个项目最重要的不是“接入大模型”，而是把广告投放归因这个复杂业务问题拆成可控的 AI 产品系统。

我做的核心取舍是：MVP 聚焦投放异常、归因核对和纯知识查询；范围外意图明确兜底；事实来源交给 RAG 和只读工具；稳定排查交给 workflow；模型负责理解和解释；风险、置信度和成本由规则与 Trace 管控。

这样这个项目在面试中能体现业务理解、AI 架构能力、产品取舍、数据指标意识和落地风险控制。
```
