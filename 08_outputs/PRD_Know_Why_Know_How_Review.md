# PRD Know Why & Know How 复盘学习文档

## 1. 文档说明

本文档基于 `03_prd/AdOps_Copilot_投放归因排障助手_PRD_v1.md` 生成，辅助参考了项目中的产品方向、竞品分析与 Prompt E2E 验证结果。

本文档的目标不是复述 PRD，而是帮助读者理解这份 PRD 背后的设计逻辑、实现路径、关键取舍、隐含假设和复盘价值。它更适合作为学习材料、项目负责人自查材料和后续迭代前的理解文档。

本文档与原 PRD 的区别：

- 原 PRD 回答“要做什么、范围是什么、交付什么”。
- 本文档回答“为什么要这样做、如何落地、哪里容易出问题、后续怎么判断做得对不对”。
- 原 PRD 面向需求交付，本文档面向理解、复盘和后续迭代。

## 2. PRD 总体理解

### 2.1 产品/功能一句话概括

AdOps Copilot 投放归因排障助手，是面向内部广告运营、客户成功和平台支持团队的 AI 辅助排障工具，用 RAG、只读工具调用、固定 workflow、证据对象和安全交付机制，帮助快速定位投放效果异常与归因数据不一致问题。

### 2.2 核心目标

PRD 中体现出的核心目标可以分为四类：

| 目标类型 | 核心目标 | 复盘理解 |
|---|---|---|
| 业务目标 | 降低投放归因排障耗时，提高内部排障会话的有效解决量 | 项目价值不在“AI 看起来聪明”，而在能否减少人工跨系统查数、查文档、对口径的时间 |
| 用户目标 | 让 AdOps、AM/CS、平台支持能更快获得可解释、可追溯的排障结论 | 用户真正需要的是“下一步怎么处理”和“依据是什么”，不是泛泛聊天 |
| 产品目标 | 将投放异常、归因差异、纯知识查询三个场景做成可控的 MVP | 通过范围收敛提高落地概率，避免一开始做成泛 Copilot |
| 技术目标 | 用 workflow 承担确定性流程，用 LLM 承担语言理解、解释和归纳，用规则层兜住风险与置信度 | 这是 AI 产品工程化落地的核心设计，减少大模型不确定性 |

### 2.3 核心用户与使用场景

核心用户包括：

- AdOps：日常排查广告消耗、转化、CPA、ROI、归因差异等问题。
- AM/CS：面对客户问题时，需要快速理解内部排查进展和可外部沟通的口径。
- 平台支持/产品运营：需要把高频问题、Badcase 和知识缺口沉淀到知识库和工具链中。
- 管理者：关注排障效率、升级质量、知识复用率和业务收益。

核心场景包括：

- 投放效果异常：例如消耗下降、安装下降、CPA 上升、ROI 异常。
- 归因数据不一致：例如平台报表、MMP 报表、客户后台数据存在差异。
- 纯知识查询：例如询问归因窗口、口径定义、SOP 或历史案例。
- Badcase 回流：例如模型回答不准、引用不完整、工具结果缺失或规则口径不一致。

这些场景重要的原因在于，它们都具有“信息分散、口径复杂、重复发生、需要证据、容易误判”的特点，适合用 AI 做理解和组织，但又不能完全交给自由生成。

### 2.4 整体需求地图

- 产品边界与范围
  - 内部排障助手

## 3. Know Why：为什么这样设计

### 3.1 模块名称：范围收敛与内部只读定位

#### 3.1.1 PRD 中的设计内容

PRD 明确将第一阶段限定为内部使用的投放归因排障助手，重点覆盖投放效果异常、归因差异核查和纯知识查询。不覆盖 SDK/API 深度排障、素材审核、客户自动回复、预算出价自动修改等高风险或跨域场景。

#### 3.1.2 为什么需要这个模块

广告排障本身涉及多个系统、多个口径和多个角色。如果第一版直接做全场景 Copilot，需求会迅速膨胀，导致权限、工具、数据、评测和责任边界都难以落地。范围收敛可以让团队先验证最核心的 AI 价值：能否基于证据帮助内部人员更快定位问题。

#### 3.1.3 为什么是当前这种设计

- 它满足了“先做高频、高价值、可验证场景”的目标。
- 它降低了外部客户沟通、账户操作、素材合规和 SDK 技术支持带来的风险。
- 它规避了模型误操作、错误承诺客户、自动修改投放策略等问题。
- 它让 MVP 的验收更清晰：只需证明排障效率和诊断质量提升，而不是证明全链路智能客服能力。

#### 3.1.4 如果不这样设计，可能会有什么问题

如果第一版做成全能助手，最可能出现的问题是功能范围难以收敛，研发排期不可控，评测集无法覆盖所有场景，安全策略也难以一次到位。尤其是客户自动回复和投放操作，如果模型给出错误建议或误触发操作，责任风险远高于内部辅助排障。

#### 3.1.5 该设计背后的隐含假设

- 内部团队有足够高频的投放和归因排障需求。
- 内部用户愿意接受“AI 辅助 + 人工确认”的协作方式。
- 排障所需的数据和知识至少可以通过只读工具或知识库获取。
- 项目第一阶段更重视可控落地，而不是对外自动化体验。

### 3.2 模块名称：总控智能体与固定 workflow 边界

#### 3.2.1 PRD 中的设计内容

PRD 将总控设计为“会话创建、权限前置、意图识别、规则归一化、workflow 分发、安全交付”的编排层。任务拆解和关键判断尽量由固定 workflow 和规则层承担，LLM 主要负责语言理解、解释归纳和结构化生成。

#### 3.2.2 为什么需要这个模块

投放排障不是开放式聊天，而是有明确检查路径和业务规则的任务。如果让模型自由决定查什么、怎么查、怎么给结论，会带来成本高、稳定性差、不可审计和难复现的问题。总控 workflow 的作用是把自然语言问题转成可控任务流。

#### 3.2.3 为什么是当前这种设计

- workflow 适合处理固定、可测试、可审计的排障步骤。
- LLM 适合处理非结构化输入、模糊表达、多语言问题和证据解释。
- 规则层适合处理权限、风险、置信度、工具约束等必须稳定的判断。
- 总控让各场景智能体保持相对独立，降低模块之间耦合。

#### 3.2.4 如果不这样设计，可能会有什么问题

如果任务拆解完全依赖大模型，排查路径可能每次不同，工具调用可能遗漏关键检查项，成本和延迟也不可控。如果完全不用 LLM，只做固定表单和规则，又难以理解运营人员的自然语言问题，也无法生成可读的解释和下一步建议。

#### 3.2.5 该设计背后的隐含假设

- 投放和归因排障存在相对稳定的 SOP 和检查清单。
- 主要意图类别可以被枚举，并且可通过测试集持续校准。
- 规则层能够拿到足够的字段来复算 risk_level_final 和 confidence_final。
- 工程团队可以实现 workflow 编排、工具注册、schema 校验和 trace 记录。

### 3.3 模块名称：RAG、工具调用与 Evidence Object

#### 3.3.1 PRD 中的设计内容

PRD 将 RAG 和数据工具定义为场景智能体获取 context 的信息源，而不是独立给结论的模块。工具和知识召回结果需要统一转成 Evidence Object，并通过 evidence_id 被 workflow_output 和 delivery_guard_output 引用。

#### 3.3.2 为什么需要这个模块

投放归因排障的结论必须可追溯。用户不会只相信“AI 说是 Postback 延迟”，而是需要看到对应报表、口径文档、时间窗口、工具结果和历史案例。Evidence Object 是把模型回答从“生成文本”拉回“证据驱动”的关键设计。

#### 3.3.3 为什么是当前这种设计

- 它满足了可解释性和可审计要求。
- 它降低了幻觉风险，因为最终结论必须引用证据。
- 它把工具结果、知识片段和历史案例统一成可复用对象。
- 它为权限控制、前端展示、Badcase 复盘和离线评测提供了共同数据结构。

#### 3.3.4 如果不这样设计，可能会有什么问题

如果没有统一证据对象，不同工具和知识库会输出各自格式，场景智能体难以稳定引用，前端也无法统一渲染。更严重的是，后续无法判断某个结论到底来自哪份文档、哪个工具、哪个时间点的数据。

#### 3.3.5 该设计背后的隐含假设

- 工具输出能够被标准化，并且每次调用都有稳定的 trace_id 和 evidence_id。
- 知识库文档具备 owner、reviewed_status、effective_date、visibility 等元数据。
- 用户愿意阅读“证据摘要 + 引用来源”，而不仅仅是结论。
- 前后端都按 evidence_refs 引用证据，而不是把证据直接混进最终文本。

### 3.4 模块名称：投放效果异常诊断

#### 3.4.1 PRD 中的设计内容

投放诊断模块用于处理消耗、展示、点击、安装、转化、CPA、ROI 等异常，通过固定指标漏斗、账户状态、投放配置、归因摘要、知识库和历史案例生成内部诊断结果。

#### 3.4.2 为什么需要这个模块

投放效果异常是广告运营最常见的问题之一，人工排查通常需要跨报表、账户配置、归因口径、投放状态和历史经验。该模块解决的是“从零散数据到排障线索”的问题。

#### 3.4.3 为什么是当前这种设计

- 指标漏斗能够把复杂问题拆成相对稳定的检查链路。
- 只读工具调用能保证安全，不会误改投放配置。
- 诊断结果输出 checked_items、likely_reasons、next_actions，可以让运营人员继续跟进。
- confidence_final 由规则层回写，可以避免模型主观给分。

#### 3.4.4 如果不这样设计，可能会有什么问题

如果只做知识问答，无法结合实时投放数据。如果只做报表告警，又缺少解释和 SOP。当前设计把结构化数据、知识库和生成式解释结合起来，适合内部排障场景。

#### 3.4.5 该设计背后的隐含假设

- 核心投放指标和账户状态可以通过 API、缓存或报表获取。
- 业务团队已有基本诊断 SOP，可以转成固定检查项。
- 异常归因不要求第一版做到完全自动判责，而是提供高质量线索。

### 3.5 模块名称：归因与数据不一致核对

#### 3.5.1 PRD 中的设计内容

归因核查模块围绕平台报表、MMP 报表、Postback 摘要和归因口径差异，使用固定 checklist 检查时区、归因窗口、事件映射、Postback 延迟、去重/再归因、隐私过滤、渠道映射和数据新鲜度。

#### 3.5.2 为什么需要这个模块

归因差异是广告平台与客户、MMP、内部报表之间最容易产生争议的场景。问题通常不是单个数值，而是多个口径、时间窗口和数据链路共同导致的差异。该模块解决的是“把争议拆成可验证检查项”的问题。

#### 3.5.3 为什么是当前这种设计

- checklist 可以确保关键排查项不遗漏。
- 证据对象可以支撑每个 likely_reason。
- LLM 只负责归纳解释和排序，避免让模型自由计算差异。
- workflow_output 与 delivery_guard_output 分离，保证内部诊断和最终展示边界清晰。

#### 3.5.4 如果不这样设计，可能会有什么问题

如果没有固定核查清单，模型可能只解释最显眼的数据差异，漏掉时区、窗口、事件映射等低频但关键的问题。如果最终交付直接使用诊断对象，也容易把内部敏感信息或不确定结论展示给不合适的用户。

#### 3.5.5 该设计背后的隐含假设

- MMP、平台和 Postback 数据可以按同一时间范围、campaign/app/event 维度被取回。
- 差异问题可以被拆成有限的高频原因集合。
- 业务团队认可 checklist 作为第一版归因核查的标准路径。

### 3.6 模块名称：安全交付与人工确认

#### 3.6.1 PRD 中的设计内容

PRD 通过 delivery_guard_v1 将场景 workflow 输出转成最终给前端渲染的结构化结果，包括 safe_summary、evidence_refs、limitations、next_actions、customer_visible_reply_allowed、requires_human_review 等字段。

#### 3.6.2 为什么需要这个模块

排障结果并不等于可直接展示或可对外沟通的内容。内部诊断可能包含敏感字段、不确定判断、权限受限信息或需要人工确认的下一步动作。安全交付层的作用是把“内部分析”转换为“可被用户安全消费的结果”。

#### 3.6.3 为什么是当前这种设计

- 它把最终前端展示字段统一到 delivery_guard_output。
- 它通过 redactions_applied 和 blocked_reasons 明确哪些内容被过滤或阻断。
- 它通过 requires_human_review 控制高风险场景转人工。
- 它避免前端直接渲染 workflow_output 中的内部字段。

#### 3.6.4 如果不这样设计，可能会有什么问题

如果没有安全交付层，前端可能直接展示诊断过程中的敏感字段、低置信结论或内部解释。这样不仅会产生合规风险，也会让用户误以为 AI 已经给出最终定论。

#### 3.6.5 该设计背后的隐含假设

- 后端能明确区分内部可见、客户可见、需脱敏、禁止展示的字段。
- delivery_guard 的规则和 Prompt 能覆盖主要高风险内容。
- 产品侧愿意牺牲部分“回答完整性”，换取安全和可审计。

### 3.7 模块名称：Badcase、Judge AI 与知识运营闭环

#### 3.7.1 PRD 中的设计内容

PRD 将 Badcase 反馈、Judge AI 离线评测、知识库更新、Prompt 版本管理和回归测试作为持续优化机制。Judge AI 不参与在线结论生成，而用于离线评估和问题归因。

#### 3.7.2 为什么需要这个模块

AI 排障产品上线后，质量问题不会一次性消失。模型可能意图识别错误、引用不准、证据不全、工具结果异常或安全交付过松。Badcase 机制解决的是“如何让系统持续变好，而不是靠一次性 Prompt 调优”的问题。

#### 3.7.3 为什么是当前这种设计

- 离线 Judge AI 可以降低线上误判影响。
- Badcase 分类可以区分问题来自知识、工具、Prompt、规则还是数据。
- 回归测试可以避免修一个案例破坏另一个案例。
- 知识运营后台让非研发角色也能参与知识质量建设。

#### 3.7.4 如果不这样设计，可能会有什么问题

如果没有反馈闭环，系统质量会停留在上线时状态。运营人员即使发现问题，也难以沉淀为可复用改进。更严重的是，团队无法证明模型版本、知识版本、Prompt 版本调整是否真的提升质量。

#### 3.7.5 该设计背后的隐含假设

- 团队有资源持续运营知识库和 Badcase。
- 评测集能够覆盖高频问题和高风险问题。
- Judge AI 的结果会被人工抽样校验，避免评价体系自身失真。

### 3.8 模块名称：指标体系与价值测算

#### 3.8.1 PRD 中的设计内容

PRD 设计了北极星指标、业务指标、AI 效果指标、安全指标、体验指标和成本指标，并给出时间节省、成本估算、Token 估算和降本策略。

#### 3.8.2 为什么需要这个模块

AI 产品容易停留在“能力展示”，但内部项目最终需要证明业务收益。指标体系的价值在于让团队能回答：用户是否真的用、是否真的省时间、是否减少升级、是否安全、成本是否可控。

#### 3.8.3 为什么是当前这种设计

- 北极星指标聚焦“有效 AI 辅助排障会话数”，比单纯调用次数更贴近价值。
- 业务指标关注人效和升级质量。
- AI 指标关注准确性、引用、完整性和安全。
- 成本指标关注单次任务费用、Token 消耗和模型分层。

#### 3.8.4 如果不这样设计，可能会有什么问题

如果只看响应速度或调用次数，系统可能看起来活跃但不解决问题。如果只看模型准确率，又无法证明业务收益。如果不看成本，后续规模化时可能出现单位会话成本不可控。

#### 3.8.5 该设计背后的隐含假设

- 系统能埋点记录会话、工具调用、人工采纳、处理时长和升级结果。
- 业务侧能提供上线前基线数据。
- 价值测算中的人工时薪、排障频次、节省比例需要在真实环境中校准。

## 4. Know How：如何落地实现

### 4.1 功能/流程名称：总控路由与权限前置

#### 4.1.1 功能目标

把用户自然语言问题转成可执行、可审计、可控风险的 routing_final，并决定是否进入投放诊断、归因核查、纯知识查询或兜底路径。

#### 4.1.2 用户操作路径

1. 用户在 Copilot 入口输入问题。
2. 用户可附带 campaign_id、app_id、advertiser_id、时间范围或截图/报表描述。
3. 系统返回初步理解、需要补充的信息或进入排查。
4. 若权限不足或场景不在范围内，系统给出安全兜底说明。

#### 4.1.3 系统处理流程

1. 创建会话并生成 trace_id。
2. 读取用户身份、角色、账户、区域和数据权限。
3. 运行意图识别 Prompt，生成 routing_candidate。
4. schema_guard 校验输出结构。
5. rule_normalizer 基于权限、实体完整性、风险信号复算 risk_level_final 与 confidence_final。
6. workflow_dispatcher 选择 selected_workflow 或 fallback。
7. 记录路由结果和决策原因。

#### 4.1.4 数据与状态变化

关键对象包括：

- session：会话、用户、入口、上下文。
- routing_candidate：模型初始路由结果。
- routing_final：规则层复算后的最终路由对象。
- permission_context：用户角色、账户权限、区域权限、可访问工具。
- tool_constraints：允许调用的工具、禁止调用的工具和原因。

#### 4.1.5 前后端/模块协作方式

- 前端需要传入用户问题、入口上下文、已选对象和可选补充信息。
- 后端负责 session、权限、schema 校验、workflow 分发和 trace。
- LLM 服务负责意图识别，不负责最终风险和权限判断。
- 权限系统提供用户角色、账户范围和敏感字段可见性。

#### 4.1.6 异常与边界情况

- 用户未提供 campaign_id 或时间范围：进入补充信息询问或知识查询兜底。
- 用户权限不足：禁止调用账户/MMP/Postback 工具。
- 用户请求 SDK/API、素材审核或改预算：识别为非当前范围。
- 用户问题同时包含知识查询和排障：优先判断是否需要数据工具，必要时拆分。
- 模型输出 schema 不合法：重试一次或走规则兜底。

#### 4.1.7 验证方式

- 用黄金测试集验证 intent、selected_workflow 和 out_of_scope 识别准确率。
- 用权限测试集验证越权工具调用为 0。
- 检查 routing_final 中 risk_level_final 和 confidence_final 是否全部来自规则层。
- 对中英文混合问题做回归测试。

### 4.2 功能/流程名称：RAG 检索到 Evidence Object

#### 4.2.1 功能目标

把知识库、报表工具、历史案例和外部口径文档转成可被场景智能体引用的 Evidence Object，为结论提供证据基础。

#### 4.2.2 用户操作路径

用户不直接操作 RAG。用户提出问题后，系统根据意图、实体和权限自动检索相关 SOP、口径、历史案例和工具数据，并在结果中展示引用。

#### 4.2.3 系统处理流程

1. 根据 routing_final 和 workflow plan 生成检索请求。
2. 权限前置过滤可检索知识与可调用工具。
3. 对知识库执行 BM25、向量、实体精确匹配和历史案例召回。
4. 通过 rerank 对召回结果排序。
5. 对工具输出执行标准化和脱敏。
6. 将知识片段、工具结果、历史案例统一包装为 Evidence Object。
7. 将 evidence_objects 写入 Evidence Store。

#### 4.2.4 数据与状态变化

关键字段包括：

- evidence_id：证据唯一标识。
- source_type：知识、工具、历史案例、报表等来源类型。
- claim_supported：证据支持的判断。
- visibility：内部可见、客户可见或受限。
- customer_visible_allowed：是否允许被外部口径引用。
- retrieved_at：检索或工具调用时间。

#### 4.2.5 前后端/模块协作方式

- 后端负责检索编排、权限过滤、证据归一化和 Evidence Store。
- 数据平台提供报表、账户、MMP、Postback 等只读接口或快照。
- 知识库系统提供文档解析、chunk、元数据和版本。
- 前端只展示 delivery_guard_output 引用到的 evidence_refs，不直接展示所有 evidence_objects。

#### 4.2.6 异常与边界情况

- 召回到过期文档：降低可信度或标记 limitations。
- 多份文档口径冲突：进入冲突提示，不能强行给唯一结论。
- 权限过滤后无证据：返回缺证据限制，不生成确定结论。
- 工具接口失败：使用缓存、ETL 快照、人工上传或降级为知识查询。
- 数据新鲜度不足：必须在证据中标注 retrieved_at 和 freshness。

#### 4.2.7 验证方式

- 检查召回结果是否来自 reviewed_status 合格的知识。
- 检查引用证据是否与结论字段一致。
- 用历史案例测试召回覆盖率和 rerank 命中率。
- 用权限样例验证不可见文档和账户数据不会进入 context。

### 4.3 功能/流程名称：投放效果异常诊断 workflow

#### 4.3.1 功能目标

在用户提出投放效果异常时，按固定漏斗和业务规则检查关键指标、账户状态、投放配置和相关知识，输出内部诊断结果与下一步建议。

#### 4.3.2 用户操作路径

1. 用户提出“某 campaign 安装下降/CPA 上升/消耗异常”等问题。
2. 系统识别实体并确认时间范围。
3. 系统展示诊断摘要、可能原因、已检查项、证据引用和建议动作。
4. 用户可继续追问某个原因、补充数据或标记 Badcase。

#### 4.3.3 系统处理流程

1. 接收 routing_final，确认 selected_workflow 为 campaign_performance_diagnosis_v1。
2. 调用 campaign metrics、account status、相关知识和历史案例。
3. 按漏斗检查展示、点击、安装、转化、成本、ROI 等指标。
4. 将工具结果转成 evidence_objects。
5. 场景 Prompt 基于证据生成 workflow_output。
6. 规则层根据证据覆盖、工具新鲜度、检查项完整度回写 confidence_final。
7. delivery_guard 生成最终 safe_summary 和 next_actions。

#### 4.3.4 数据与状态变化

涉及的数据包括：

- campaign_id、app_id、advertiser_id、date_range。
- spend、impressions、clicks、installs、conversions、CPA、ROI 等指标。
- account_status、budget_status、delivery_status。
- workflow_output 中的 checked_items、likely_reasons、excluded_reasons、required_followups。
- confidence_rule_inputs 和最终 confidence_final。

#### 4.3.5 前后端/模块协作方式

- 前端负责呈现诊断卡片、证据引用、限制说明和追问入口。
- 后端负责工具调用、workflow 编排和 delivery_guard。
- 数据平台负责指标与状态接口。
- 知识运营负责维护投放异常 SOP 和历史案例标签。

#### 4.3.6 异常与边界情况

- campaign_id 不存在或用户无权限。
- 用户只描述现象但缺少时间范围。
- 指标数据为空或延迟。
- 工具返回的当前状态与历史快照冲突。
- 多个可能原因并存，无法给出单一结论。
- 模型想输出没有证据支持的原因，需要被规则或 delivery_guard 拦截。

#### 4.3.7 验证方式

- 用历史投放异常案例验证 likely_reasons 命中率。
- 验证 checked_items 是否覆盖固定漏斗。
- 验证低证据覆盖时不会输出高置信结论。
- 验证最终前端展示来自 delivery_guard_output，而不是 workflow_output。

### 4.4 功能/流程名称：归因差异核查 workflow

#### 4.4.1 功能目标

在平台、MMP、Postback 或客户后台数据不一致时，按固定归因 checklist 核查差异原因，并输出可追溯的内部诊断结果。

#### 4.4.2 用户操作路径

1. 用户提出“平台安装 1250，MMP 安装 900，为什么差这么多”等问题。
2. 用户提供或系统识别 campaign、app、event、时间范围和对比口径。
3. 系统返回差异摘要、核查项结果、可能原因、证据引用、限制和下一步建议。
4. 用户可要求展开某个核查项或提交 Badcase。

#### 4.4.3 系统处理流程

1. selected_workflow 进入 attribution_discrepancy_v1。
2. workflow 根据 checklist 生成工具计划。
3. 调用平台报表、MMP 报表、Postback 摘要、归因政策和知识库。
4. 计算差异比例、数据新鲜度、Postback 延迟等确定性字段。
5. 将工具结果包装为 evidence_objects。
6. 场景 Prompt 输出 checked_items、likely_reasons、excluded_reasons、required_followups 和 confidence_rule_inputs。
7. 规则层回写最终 confidence_final。
8. delivery_guard 生成最终 safe_summary、evidence_refs、limitations 和 next_actions。

#### 4.4.4 数据与状态变化

关键数据包括：

- platform_installs、mmp_installs、postback_success_rate、postback_delay。
- attribution_window、timezone、event_mapping、dedup_rule、reattribution_rule。
- checklist 状态：passed、failed、unknown、not_applicable。
- evidence_objects 与 delivery_guard_output.evidence_refs 的引用关系。

#### 4.4.5 前后端/模块协作方式

- 前端展示差异卡片、核查清单和证据引用。
- 后端负责工具计划、差异计算、证据归一化和安全交付。
- MMP 或数据平台提供只读报表与 Postback 摘要。
- 业务配置系统提供归因窗口、事件映射和渠道映射。

#### 4.4.6 异常与边界情况

- 平台和 MMP 的时区不同，导致表面差异。
- 数据仍在回流期，MMP 或 Postback 尚未完全入库。
- 事件映射口径不一致，安装和转化事件被混用。
- 用户提供的数字来自截图或手工报表，无法验证来源。
- 工具 API 不可用，只能基于缓存或人工上传数据做低置信分析。
- 涉及客户后台数据但系统无权限读取。

#### 4.4.7 验证方式

- 用归因黄金测试集验证 checklist 完整率。
- 检查 likely_reasons 是否必须引用 evidence_id。
- 检查差异计算字段由系统计算，而不是模型生成。
- 检查低新鲜度或缺工具场景是否输出 limitations。

### 4.5 功能/流程名称：纯知识查询与引用回答

#### 4.5.1 功能目标

在用户询问口径定义、SOP、产品规则或历史案例时，通过 RAG 生成带引用的知识回答，不调用账户、MMP、Postback 等敏感数据工具。

#### 4.5.2 用户操作路径

1. 用户提出“归因窗口是什么”“Postback 延迟一般怎么判断”等问题。
2. 系统返回知识回答、引用来源、适用范围和不确定限制。
3. 用户可继续追问或要求转为具体 campaign 排障。

#### 4.5.3 系统处理流程

1. 总控识别为 knowledge_lookup。
2. rule_normalizer 禁止敏感数据工具调用。
3. RAG 检索相关 SOP、口径文档和历史案例。
4. 生成带 citations 的回答。
5. delivery_guard 校验是否包含越权信息或未引用结论。

#### 4.5.4 数据与状态变化

关键对象包括：

- knowledge_query。
- retrieved_chunks。
- citations。
- limitations。
- answer_scope。

#### 4.5.5 前后端/模块协作方式

- 前端展示引用来源、文档标题、更新时间和适用范围。
- 后端负责检索、引用校验和安全过滤。
- 知识库运营负责文档质量、语言版本和过期提醒。

#### 4.5.6 异常与边界情况

- 问题看似知识查询，但包含具体账户或 campaign。
- 检索结果互相冲突。
- 用户请求客户可见口径，但知识仅内部可见。
- 查询内容不在知识库范围内。

#### 4.5.7 验证方式

- 验证回答引用覆盖率。
- 验证无证据时是否明确说明无法确认。
- 验证纯知识查询不会调用敏感工具。
- 验证中英文混合问题能召回对应语言或跨语言文档。

### 4.6 功能/流程名称：Delivery Guard 最终交付

#### 4.6.1 功能目标

将 routing_final、workflow_output 和 evidence_objects 转换成前端可渲染、权限安全、风险可控的 delivery_guard_output。

#### 4.6.2 用户操作路径

用户看到的是最终卡片，包括安全摘要、证据引用、限制说明、下一步动作和是否需要人工审核，而不是原始 workflow_output。

#### 4.6.3 系统处理流程

1. 接收 routing_final、workflow_output、evidence_objects 和 permission_context。
2. 检查 risk_level_final、confidence_final 和是否需要人工审核。
3. 删除或脱敏不应展示的信息。
4. 将内部诊断摘要改写为 safe_summary。
5. 只保留可引用 evidence_refs。
6. 输出 delivery_guard_v1 schema。

#### 4.6.4 数据与状态变化

最终前端展示结构包括：

- schema_version。
- trace_id。
- delivery_type。
- risk_level_final。
- confidence_final。
- safe_summary。
- evidence_refs。
- redactions_applied。
- blocked_reasons。
- limitations。
- next_actions。
- customer_visible_reply_allowed。
- requires_human_review。

#### 4.6.5 前后端/模块协作方式

- 后端提供 delivery_guard_output 作为唯一前端渲染来源。
- 前端根据 delivery_type、risk_level_final、requires_human_review 展示不同状态。
- 权限系统提供脱敏和可见性规则。
- 审计系统记录被过滤字段和阻断原因。

#### 4.6.6 异常与边界情况

- workflow_output 中含有敏感字段。
- evidence_refs 指向用户无权查看的 evidence_objects。
- confidence_final 过低但模型摘要语气过于确定。
- 用户要求对外发送给客户，但 customer_visible_reply_allowed 为 false。
- 高风险场景未触发人工审核。

#### 4.6.7 验证方式

- 验证前端只消费 delivery_guard_output。
- 用安全测试集验证敏感字段脱敏。
- 验证 blocked_reasons 与 redactions_applied 可审计。
- 验证 evidence_refs 都能在 Evidence Store 找到对应 evidence_id。

### 4.7 功能/流程名称：Badcase 与 Judge AI 回流

#### 4.7.1 功能目标

把线上错误、低质量回答和人工反馈转成可处理的改进任务，并通过离线评测验证知识、Prompt、规则或工具调整是否有效。

#### 4.7.2 用户操作路径

1. 用户在结果卡片上标记不准确、不完整、不可用或需人工处理。
2. 用户可补充正确原因或期望结果。
3. 运营或产品在后台分类 Badcase。
4. 调整知识、Prompt、规则或工具配置。
5. 回归测试通过后发布新版本。

#### 4.7.3 系统处理流程

1. 收集 trace、输入、routing_final、workflow_output、delivery_guard_output 和用户反馈。
2. Judge AI 离线评估问题类型。
3. 人工复核并确认归因。
4. 分派给知识运营、产品、算法或研发。
5. 更新知识库、Prompt 或规则。
6. 运行黄金集和 Badcase 回归。

#### 4.7.4 数据与状态变化

关键状态包括：

- badcase_status：new、triaged、fixed、verified、closed。
- issue_type：intent、retrieval、tool、prompt、schema、safety、business_rule。
- fix_version：知识版本、Prompt 版本、规则版本或工具版本。
- eval_result：通过、失败、需人工复核。

#### 4.7.5 前后端/模块协作方式

- 前端提供反馈入口。
- 后端保存完整 trace。
- 运营后台支持 Badcase 分类、处理和回归状态。
- Judge AI 只做离线辅助判断。

#### 4.7.6 异常与边界情况

- 用户反馈不具体，无法判断问题来源。
- Judge AI 和人工判断不一致。
- 修复某类 Badcase 后影响其他场景。
- 知识修复后未重新索引或未通过 review。

#### 4.7.7 验证方式

- 验证 Badcase 能关联完整 trace。
- 验证修复后对应案例通过回归测试。
- 验证新版本不会降低关键黄金集指标。
- 验证人工抽检机制存在，避免 Judge AI 自我循环。

## 5. 关键业务规则解析

| 业务规则 | 规则说明 | 为什么需要 | 如何实现 | 风险/注意点 |
|---|---|---|---|---|
| 第一阶段内部只读 | 系统只辅助内部排障，不自动修改预算、出价、账户配置 | 降低误操作和责任风险 | 工具注册层只开放 read-only tool，delivery_guard 阻断操作建议 | 用户可能期望系统直接修复问题，需要交互上说明边界 |
| 范围限定 | 覆盖投放异常、归因差异、纯知识查询，不覆盖 SDK/API、素材审核、客户回复自动化 | 控制 MVP 复杂度 | 总控意图识别和 out_of_scope fallback | 用户问题可能混合多个场景，需要拆分或追问 |
| 权限前置 | 工具调用和知识检索前先检查用户权限 | 防止越权读取账户、MMP 或客户数据 | permission_context + RBAC/ABAC + tool_constraints | 权限模型过粗会影响可用性，过细会增加实现成本 |
| risk_level_final 规则层复算 | 最终风险等级不由模型自由判断 | 风险判断必须稳定、可审计 | 基于敏感数据、客户可见、权限缺口、操作风险等规则计算 | 规则过保守会增加人工审核，过宽会带来安全风险 |
| confidence_final 规则层回写 | 最终置信度基于证据覆盖、工具新鲜度、检查项完整度等计算 | 避免模型主观打分 | workflow_output 输出 confidence_rule_inputs，规则层回写 confidence_final | 公式需要真实数据校准，不能只停留在 PRD |
| 证据驱动结论 | 诊断结论必须引用 evidence_id | 降低幻觉，提高可解释性 | workflow_output 引用 evidence_objects，delivery_guard 输出 evidence_refs | 证据召回错误会导致“有引用但结论错” |
| 前端只渲染 delivery_guard_output | workflow_output 不是最终 UI 对象 | 防止内部字段、敏感信息和不确定结论外泄 | API 合约规定前端唯一消费 delivery_guard_v1 | 研发实现时需要严格区分中间对象和交付对象 |
| 固定归因 checklist | 归因核查必须检查时区、窗口、事件映射、延迟、去重等项目 | 防止遗漏高频原因 | workflow 编排固定检查项，模型只解释排序 | checklist 需要随业务演进更新 |
| 纯知识查询不调用敏感工具 | knowledge_lookup 只走 RAG，不读取账户/MMP/Postback 数据 | 降低低风险场景成本和权限风险 | selected_workflow 约束 tool_constraints | 某些问题需要从知识查询升级到具体排障 |
| 工具失败降级 | API 不可用时可使用缓存、ETL、人工上传或仅知识回答 | 提升可用性，避免流程完全中断 | 工具层返回 freshness、fallback_source 和 limitations | 降级结果必须降低置信度并明确限制 |
| Badcase 先评估再发布 | Badcase 修复必须经过回归验证 | 防止 Prompt 或知识修复引入新问题 | eval set + Judge AI + 人工抽检 + 版本管理 | 如果缺少负责人，闭环容易停在“收集反馈” |
| 客户可见严格控制 | 默认内部诊断不等于客户可见回复 | 避免泄露内部数据或不成熟判断 | customer_visible_reply_allowed 和 requires_human_review 控制 | AM/CS 仍可能复制内部内容，需要 UI 提醒 |

## 6. 核心流程复盘

### 6.1 流程名称：端到端投放归因排障流程

#### 流程目标

把用户的自然语言问题转成可控的排障流程，并输出有证据、有边界、有下一步建议的最终结果。

#### 流程步骤

1. 用户发起问题并创建会话。
2. 系统读取用户身份、权限和入口上下文。
3. 总控识别意图、实体、风险信号。
4. 规则层复算 routing_final。
5. workflow_dispatcher 选择场景 workflow。
6. 场景 workflow 调用 RAG 和只读工具。
7. 工具和知识结果写入 Evidence Store。
8. 场景 Prompt 生成 workflow_output。
9. 规则层回写 confidence_final。
10. delivery_guard 生成最终前端展示结果。
11. 用户采纳、追问、转人工或提交 Badcase。

#### 流程背后的设计原因

这个流程把“不确定的语言理解”和“确定的排障执行”拆开处理。LLM 不是全流程主宰，而是参与理解与表达；规则和 workflow 负责安全、稳定、成本和审计。

#### 流程中的关键判断点

- 用户问题是否在当前范围内。
- 是否有足够实体进入排障。
- 用户是否有对应数据权限。
- 工具结果是否新鲜且完整。
- 证据是否足够支撑结论。
- 是否需要人工审核。

#### 流程风险

- 意图识别错误导致进入错误 workflow。
- 工具数据缺失但模型仍输出确定结论。
- 前端误用 workflow_output。
- 用户过度信任 AI 摘要，忽略 limitations。

### 6.2 流程名称：归因差异核查流程

#### 流程目标

将平台、MMP、Postback 和客户口径之间的数据差异拆成一组可验证的归因检查项。

#### 流程步骤

1. 识别 campaign、app、event、date_range 和对比数据源。
2. 拉取平台报表、MMP 报表和 Postback 摘要。
3. 检查时区和统计窗口。
4. 检查归因窗口和事件映射。
5. 检查 Postback 延迟、成功率和数据新鲜度。
6. 检查去重、再归因、隐私过滤和渠道映射。
7. 输出 checked_items、likely_reasons 和 required_followups。
8. delivery_guard 生成最终安全摘要和证据引用。

#### 流程背后的设计原因

归因问题的关键不是“让 AI 猜原因”，而是把争议转成标准核查项，逐项排除或确认。这样更符合广告业务里的协作方式，也更容易被运营和客户成功团队接受。

#### 流程中的关键判断点

- 两边数据是否处于同一时区和统计周期。
- 事件是否同名同义。
- 数据是否仍在回流。
- 是否存在口径或配置差异。
- 是否缺少关键数据源。

#### 流程风险

- 用户提供的数据来源不可信。
- 工具拿到的是延迟数据。
- checklist 不能覆盖某些特殊 MMP 规则。
- 多个原因同时存在，容易被误读为单一原因。

### 6.3 流程名称：纯知识查询流程

#### 流程目标

让内部用户快速获取口径、SOP 和历史案例，同时避免不必要的数据工具调用。

#### 流程步骤

1. 识别意图为 knowledge_lookup。
2. 禁止敏感账户和归因工具调用。
3. 检索知识库、SOP 和历史案例。
4. 检查文档权限、版本和冲突。
5. 生成带引用回答。
6. delivery_guard 输出 safe_summary、citations 和 limitations。

#### 流程背后的设计原因

纯知识问题不需要高成本、高权限的数据调用。把知识查询单独拆出来，可以降低延迟、成本和安全风险，也能提高口径复用。

#### 流程中的关键判断点

- 问题是否真的不需要具体数据。
- 检索结果是否有足够引用。
- 文档是否过期或冲突。
- 用户是否请求客户可见口径。

#### 流程风险

- 知识库内容不完整导致回答泛化。
- 文档冲突但系统没有提示。
- 用户把知识回答误用于具体客户场景。

### 6.4 流程名称：Badcase 反馈与迭代流程

#### 流程目标

把线上质量问题转成可追踪、可修复、可验证的产品和技术改进。

#### 流程步骤

1. 用户或运营提交 Badcase。
2. 系统关联完整 trace 和输出对象。
3. Judge AI 离线辅助判断问题类型。
4. 人工复核并归因到知识、Prompt、工具、规则或数据。
5. 对应负责人修复。
6. 回归测试通过后发布。
7. 指标看板跟踪修复效果。

#### 流程背后的设计原因

AI 系统的质量提升依赖持续反馈，而不是一次性上线。Badcase 流程让“用户觉得不准”变成“可以定位、可以修、可以验证”的工程问题。

#### 流程中的关键判断点

- Badcase 是否有足够上下文。
- 问题来源是检索、工具、Prompt、规则还是数据。
- 修复是否影响其他场景。
- 是否需要更新黄金测试集。

#### 流程风险

- 反馈入口有了，但缺少运营处理机制。
- Judge AI 评价结果被过度信任。
- 修复没有版本管理，难以回溯。

## 7. PRD 中的隐含假设与待确认问题

### 7.1 隐含假设

| 隐含假设 | 影响范围 | 如果假设不成立会怎样 |
|---|---|---|
| 内部排障场景足够高频 | 项目价值、北极星指标 | 使用量不足，无法证明 ROI |
| 数据工具能提供稳定只读接口 | 投放诊断、归因核查 | workflow 无法自动获取证据，只能退化为知识查询 |
| 权限系统可以细分到账号、区域、字段和工具 | 安全交付、工具调用 | 越权风险或可用性下降 |
| 业务 SOP 可以结构化成 checklist | workflow 设计 | 排查流程仍依赖经验，难以自动化 |
| 知识库有 owner 和 review 机制 | RAG 质量 | 召回过期或冲突文档，回答质量不稳定 |
| 用户愿意提供反馈或标记 Badcase | 持续优化 | 系统难以形成质量闭环 |
| 规则层可以量化 confidence_final | 结果可信度 | 置信度可能退化为伪精确字段 |
| 管理者认可时间节省的测算方式 | 价值证明 | 项目收益难以被业务方接受 |

### 7.2 待确认问题

| 问题 | 关联模块 | 为什么需要确认 | 建议找谁确认 |
|---|---|---|---|
| 平台报表、MMP 报表、Postback 摘要是否已有 API | 工具调用、归因核查 | 决定 MVP 能否自动完成关键检查 | 数据平台、后端研发、MMP 对接负责人 |
| 核心排障 SOP 是否已有稳定版本 | 投放诊断、归因核查 | 决定 workflow checklist 是否可靠 | AdOps 负责人、平台支持负责人 |
| 权限系统是否支持字段级或工具级控制 | 安全与合规 | 决定越权风险和脱敏实现难度 | 后端研发、安全/合规负责人 |
| customer_visible_reply_allowed 的实际业务边界是什么 | Delivery Guard | 决定哪些摘要能被 AM/CS 对外引用 | 法务、客户成功负责人、业务负责人 |
| 北极星指标的有效会话如何判定 | 指标体系 | 决定项目价值是否可统计 | 产品、数据分析、业务负责人 |
| 上线前人工排障耗时基线如何采集 | 价值测算 | 决定节省时间公式能否落地 | 业务运营、数据分析 |
| Badcase 的处理 SLA 和 owner 是谁 | 运营闭环 | 决定反馈是否真的能推动系统优化 | 产品、运营、算法、知识库负责人 |
| 国内模型供应商采购、部署和合规边界 | 模型选型 | 决定模型方案能否真实使用 | 采购、法务、算法平台 |

## 8. 风险、取舍与替代方案

### 8.1 主要风险

产品风险：

- 用户把 AI 诊断当成最终结论，而不是辅助排障线索。
- 第一阶段范围被不断扩展，导致 MVP 变重。
- 用户反馈入口使用率低，Badcase 闭环无法运转。

技术风险：

- 工具 API 不稳定，导致 workflow 频繁降级。
- RAG 召回到错误或过期文档。
- schema 对齐在研发实现中被破坏，前端误用中间对象。
- 模型在中英文混合、广告术语和缩写上理解不稳定。

体验风险：

- 结果卡片信息过多，用户抓不住下一步动作。
- 限制说明太多，用户觉得系统“不敢回答”。
- 高风险转人工过于频繁，影响效率。

数据风险：

- 多数据源口径不一致，系统难以判断哪个是基准。
- 数据新鲜度和延迟未明确标记。
- 权限过滤后证据不足，导致结果不可用。

运营风险：

- 知识库无人维护或 owner 不清。
- Badcase 处理周期过长。
- 评测集无法跟上真实业务变化。

### 8.2 当前方案的取舍

| 取舍维度 | 当前方案倾向 | 设计理由 | 代价 |
|---|---|---|---|
| 简单性 vs 完整性 | 优先范围收敛 | 先验证投放归因排障核心价值 | SDK/API、素材审核等需求需后续 PRD |
| 用户体验 vs 实现成本 | 优先内部用户和卡片化结果 | 内部场景容错高，更容易收集反馈 | 对外客户体验暂不完整 |
| 短期上线 vs 长期扩展 | 用固定 workflow 做 MVP | 更稳定、更可测试 | 需要后续维护 checklist 和规则 |
| 自动化能力 vs 人工干预 | 高风险转人工 | 降低安全与责任风险 | 某些场景无法端到端自动解决 |
| 灵活性 vs 规则稳定性 | 风险和置信度由规则层控制 | 保证可审计和一致性 | 规则设计和维护成本上升 |
| 模型能力 vs 成本控制 | 国内模型优先、模型分层 | 降低成本和合规风险 | 需要验证复杂推理与英文业务术语能力 |

### 8.3 可能的替代方案

| 替代方案 | 优点 | 缺点 | 当前 PRD 可能未选择的原因 |
|---|---|---|---|
| 完全自由 Agent 自主规划工具 | 灵活，能处理复杂开放问题 | 不稳定、成本高、难审计、难评测 | 第一阶段更重视确定性和可控落地 |
| 纯规则排障系统 | 稳定、低成本、可审计 | 难理解自然语言，解释能力弱，知识复用差 | 用户问题表达复杂，需要 LLM 做理解和总结 |
| 只做知识库问答 | 上线快，风险低 | 无法结合实时投放和归因数据 | 排障核心价值来自数据证据和业务判断 |
| 直接做客户可见 AI 客服 | 对外价值更显性 | 合规、安全、责任和口径风险高 | 内部排障是更可控的第一阶段 |
| 先做 SDK/API 深度排障 | 技术壁垒高 | 涉及日志、集成、客户端环境和工程支持，复杂度更高 | 当前项目定位是广告投放归因业务排障 |

## 9. 面向实现的拆解建议

### 9.1 MVP 范围

第一版必须做：

- 总控路由、权限前置、schema_guard、rule_normalizer。
- 投放异常诊断 workflow 的最小漏斗检查。
- 归因差异核查 workflow 的固定 checklist。
- 纯知识查询 workflow。
- RAG 最小链路：文档解析、chunk、权限过滤、召回、rerank、引用。
- 标准 Evidence Object 和 Evidence Store。
- delivery_guard_v1 作为前端唯一展示输出。
- 基础结果卡片和 evidence_refs 展示。
- Prompt E2E 验证集和核心安全测试集。
- 基础 Badcase 提交和 trace 记录。

### 9.2 可后置范围

可以后置：

- 更完整的 Judge AI 自动评分后台。
- 高级成本看板和模型 A/B。
- 多语言知识自动对齐和翻译工作流。
- 复杂客户可见回复生成。
- SDK/API 排障、素材审核和自动操作类功能。
- 大规模知识运营工作台的高级权限和流程。

### 9.3 推荐开发顺序

1. 先定 schema：routing_final、workflow_output、evidence_objects、delivery_guard_output。
2. 搭建权限和工具注册框架，先用 mock 工具跑通。
3. 实现 RAG 最小检索链路和 Evidence Store。
4. 实现总控 workflow 和 out_of_scope/knowledge_lookup 兜底。
5. 实现归因差异 workflow，因为 checklist 最清晰、最适合先验证。
6. 实现投放异常诊断 workflow。
7. 接入 delivery_guard 和前端结果卡片。
8. 补 Prompt E2E、黄金测试集和安全测试。
9. 上线灰度，收集真实 Badcase。
10. 迭代知识、规则、Prompt、工具和指标看板。

### 9.4 关键验收标准

- 前端最终展示只来自 delivery_guard_output。
- 所有结论都有 evidence_refs 或明确 limitations。
- 权限不足时不会调用受限工具。
- risk_level_final 和 confidence_final 均由规则层或 workflow 规则回写，不由模型自由输出。
- attribution_discrepancy_v1 覆盖固定 checklist。
- knowledge_lookup 不调用敏感数据工具。
- 工具失败时能降级并降低置信度。
- Prompt E2E 测试覆盖意图、风险、置信度、证据引用和 schema。
- Badcase 能关联 trace 并进入处理流程。

## 10. 面向我个人学习的复盘总结

### 10.1 这份 PRD 最值得学习的地方

1. 把 AI 能力拆成 LLM、workflow、RAG、工具、规则、安全交付，而不是笼统写“调用大模型”。
2. 把范围控制在内部只读排障，用可控场景验证 AI 产品价值。
3. 用 Evidence Object 把工具、知识和最终回答连接起来，解决可追溯问题。
4. 明确 routing_final、workflow_output、delivery_guard_output 三层对象，避免中间诊断和最终展示混用。
5. 将 Badcase、Judge AI、Prompt E2E 和知识运营纳入产品设计，体现 AI 产品上线后的持续治理。

### 10.2 这份 PRD 容易忽略的地方

1. 权限和安全不是后置合规事项，而是总控、RAG、工具和前端展示都要共同遵守的约束。
2. confidence_final 不能让模型凭感觉输出，必须有证据覆盖、工具新鲜度和规则计算依据。
3. RAG 的价值不只是回答问题，而是给场景 workflow 提供可引用 context。
4. 纯知识查询和具体排障必须区分，否则会带来不必要的工具成本和权限风险。
5. 价值测算必须有埋点和基线，否则“节省时间”只能停留在估算。

### 10.3 我下次写类似 PRD 时应该注意什么

- 先写清楚产品边界，尤其是“不做什么”和“为什么暂时不做”。
- 对 AI 模块要写清楚输入、输出、变量来源、执行主体和失败兜底。
- 对每个模型输出字段都要问：它是模型生成、规则计算、工具返回，还是人工回写。
- 业务流程图不仅要画节点，还要能解释每个节点的责任边界。
- 指标要同时覆盖业务效果、AI 效果、安全、体验和成本。
- 对成本、置信度、风险等级这类字段，要给出公式、来源和验证方式。
- 将 Prompt、工具、RAG、评测和运营闭环放在核心需求中，而不是附件里轻描淡写。

### 10.4 用一句话总结这个 PRD 的核心设计逻辑

这份 PRD 的核心逻辑是：用固定 workflow 和规则层兜住投放归因排障的稳定性、安全性和可审计性，用 LLM 处理自然语言理解和证据解释，用 RAG 与只读工具提供可信 context，最终通过 delivery_guard 把内部诊断转成可安全消费的前端结果。
