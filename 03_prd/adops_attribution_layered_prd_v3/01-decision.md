## 1. 项目摘要

AdOps Copilot 投放归因排障助手是一款面向移动互联网广告团队的内部 AI 排障助手，优先服务广告运营、客户成功、广告优化师和技术支持等角色。产品嵌入广告后台或团队协作入口，通过 RAG 知识检索、历史工单相似案例检索、广告指标拆解、只读数据工具调用和人工确认闭环，帮助一线团队更快处理投放效果异常和归因数据不一致问题。

本期 PRD 聚焦第一阶段 MVP：投放效果异常诊断、归因与数据不一致核对、知识检索、证据管理、人工反馈与 Badcase 回流。本期不把 AdOps Copilot 做成开放式全能 AI，也不直接覆盖 SDK/API 深度排障、素材审核自动判定、客户英文回复自动发送或广告预算自动调整。

项目需要解决的核心问题不是“增加一个聊天机器人”，而是把资深广告运营的隐性排障经验、分散的 SOP/平台/MMP/历史工单知识、以及多系统数据核对流程，转化为可复用、可追溯、可评测的产品化工作流。

## 2. 背景与问题

### 2.1 行业与业务背景

移动互联网广告链路包含广告主预算、广告计划、素材、定向、出价、广告请求、曝光、点击、下载、注册、付费、MMP 归因、平台报表和广告主 BI 等多个环节。一个指标异常往往不是单点问题，而是由流量结构、素材表现、归因窗口、事件映射、回传延迟、数据刷新、渠道口径或产品漏斗共同影响。

在真实 AdOps 工作中，投放和归因问题通常需要跨广告后台、MMP、BI、SDK/Postback 摘要、平台文档、内部 SOP、历史工单和团队聊天记录排查。一线人员面临的问题不是没有信息，而是信息分散、口径不统一、经验难复用、客户等待时间长。

### 2.2 用户、场景与痛点

| 用户/角色 | 高频场景 | 当前做法 | 核心痛点 | 影响 |
| --- | --- | --- | --- | --- |
| 广告运营 | 消耗下降、CPA/CPI 上升、转化掉量、国家/素材/广告位维度异常 | 手工查广告后台、BI、历史聊天和 SOP | 系统多、指标多、缺少标准检查顺序 | 诊断耗时长，重复问资深同事 |
| 客户成功/AM | 客户追问平台与 MMP 数据差异、要求解释投放异常 | 转述给运营/技术，再整理客户回复 | 技术细节不够熟，回复专业度不稳定 | 响应慢，客户信任下降 |
| 广告优化师 | 判断预算是否加减、素材是否疲劳、渠道是否异常 | 手工拆 CTR、CVR、CPI、CPA、ROAS 等指标 | 数据维度多，难判断变化驱动因素 | 优化动作滞后，预算浪费 |
| 技术支持 | 确认 postback、事件回传、归因配置、数据链路状态 | 反复索要 campaign、app、event、日志截图 | 上游输入不完整，大量简单问题占用技术时间 | 复杂问题被简单问题挤占 |
| 产品/运营管理者 | 维护指标口径、SOP、排障经验、培训新人 | 靠文档和人工培训 | 经验难沉淀，质量依赖个人 | 新人上手慢，服务质量波动 |

### 2.3 核心问题

**问题一：知识分散，检索成本高。** 投放 SOP、MMP 归因文档、平台帮助中心、SDK/Postback 说明、审核政策、历史工单和团队聊天记录分散在多个系统中。一线人员遇到问题时，往往不知道该先查哪份材料。

**问题二：投放诊断依赖资深经验。** CPA 上涨可能来自 CPM 上涨、CTR 下降、点击到安装 CVR 下降、安装到付费 CVR 下降、素材疲劳、流量结构变化、归因延迟、MMP 回传失败或产品漏斗异常。新人缺少资深运营的拆解路径。

**问题三：归因差异解释难标准化。** 平台、MMP 和广告主 BI 对时间口径、归因窗口、事件定义、去重、再归因、SKAN、SAN 和 postback 延迟的处理不同。简单说“口径不同”无法支撑客户沟通，也无法指导下一步核查。

**问题四：跨团队流转慢。** 客户问题经常经历 AM、运营、数据、技术支持、产品、媒体方多轮转述。信息丢失、截图反复索要、字段不完整会显著拉长响应时间。

### 2.4 机会判断

| 判断 | 结论 |
| --- | --- |
| 为什么现在做 | 移动广告投放和归因问题高频、重复、链路长，团队已有 SOP、平台文档、MMP 文档、历史工单和后台数据基础，适合用 AI 做检索、归纳、排障流程编排和证据汇总。 |
| 不做的代价 | 继续依赖资深人员手工排查，新人培训慢，客户响应慢，重复问题无法沉淀为可复用资产。 |
| 与业务目标关系 | 提升广告运营和客户成功效率，减少低质量升级，提升客户解释一致性和团队经验资产化能力。 |
| 竞品空白 | AppsFlyer AI Assistant 更靠近 MMP 内数据分析，Optmyzr 更靠近 PPC 账户优化，Zendesk 更靠近客服流程；市场缺少同时覆盖移动广告投放、归因、数据口径、内部 SOP 和客户服务流转的垂直 AdOps Copilot。 |

## 3. 目标与指标

### 3.1 产品目标

1. 让一线团队在一个入口完成投放异常和归因差异问题的初步诊断。
2. 让每个 AI 诊断结论都具备证据、引用、口径说明、置信状态和人工确认路径。
3. 将资深运营的排障经验固化为场景化 workflow，而不是依赖模型自由推理。
4. 将用户反馈、人工修正和线上错误沉淀为 Badcase、知识库修订和回归评测样本。
5. 形成一份能支撑 AI 产品经理求职表达的可信项目资产，体现业务理解、AI 落地、数据评估和风险控制。

### 3.2 北极星指标

北极星指标：**有效 AI 辅助投放归因排障会话数**。

一次会话被计入该指标，需要同时满足：

| 条件 | 说明 |
| --- | --- |
| 真实场景 | 用户提出真实投放异常、归因差异或相关口径问题。 |
| 权限合法 | 用户具备对应账户、客户、campaign 或 app 的数据访问权限。 |
| 证据充分 | 输出引用至少一个可靠知识来源或只读工具结果。 |
| 形成动作 | 用户采纳、复制内部摘要、执行下一步检查、创建人工升级摘要或标记为有效。 |
| 可追溯 | 会话 trace、工具结果、证据对象、Prompt/知识版本和反馈事件完整保存。 |

单纯聊天次数、模型调用次数、页面访问次数、权限拦截次数、无证据回答不计入北极星指标。

### 3.3 成功指标

以下目标分为两类：**真实材料中的方向性估算**和**本 PRD 为 MVP 设计的灰度验收目标**。前者不能包装成已验证结果，后者用于产品上线判断。

| 指标 | 当前值/基线 | MVP 目标值 | 口径 | 归因方式 |
| --- | --- | --- | --- | --- |
| AI 辅助解决率 | 待基线采样；方向材料估算常见问题自助解决率可达 40%-60% | 灰度 4 周后达到 50% 以上 | 无需升级其他团队即可形成可执行下一步动作的会话数 / 投放归因排障会话总数 | Copilot trace、工单状态、人工抽检 |
| 人工采纳率 | 待基线采样 | 灰度期核心用户采纳或部分采纳达到 60% | 用户点击采纳、复制摘要、按建议执行或标记部分有效的会话数 / 已输出诊断会话数 | 前端反馈事件、人工抽检 |
| 平均初步诊断时长 | 待采样；旧流程常见为分钟到小时级 | 同类问题 P50 下降 40% | 从用户提交问题到形成第一版可执行排查路径的时长 | 会话日志与工单基线对比 |
| 归因核查完整率 | 待建立黄金集 | 黄金集达到 85% 以上 | 覆盖时区、归因窗口、事件映射、去重/再归因、postback、数据刷新、渠道映射等关键项的比例 | Golden Dataset、人工评审 |
| 投放异常检查完整率 | 待建立黄金集 | 黄金集达到 85% 以上 | 覆盖预算、出价、库存、素材、CTR/CVR/CPI/CPA 拆解、转化链路、归因延迟等关键项的比例 | Golden Dataset、人工评审 |
| 引用/证据覆盖率 | 待基线采样 | 可交付回答达到 90% 以上 | 带有可追溯 evidence object 的回答数 / 可交付回答数 | Evidence Store |
| 错误建议率 | 待抽检 | P0/P1 错误为 0，P2 错误逐周下降 | 主要结论错误、证据不足、越权或漏掉关键核查项的会话数 / 已抽检会话数 | Badcase 后台、人工抽检 |
| 越权阻断率 | 待压测 | 权限压力集 100% 阻断 | 权限不足或敏感请求被正确拦截的样本数 / 权限压力集样本数 | 安全评测集、审计日志 |
| Badcase 修复周期 | 待建立流程 | 核心场景小于 7 个工作日 | Badcase 从确认到修复版本通过回归的工作日 | Badcase 队列、版本记录 |

为让 PRD 可推演，本项目允许使用模拟生产数据做产品设计样例，但不得把样例写成真实业务成果。

| 模拟对象 | 样例设定 | 用途 | 风险说明 |
| --- | --- | --- | --- |
| 灰度团队 | 8 名 AdOps/AM/技术支持核心用户 | 验证 workflow、反馈和看板是否可用 | 非真实组织承诺 |

## 4. 非目标

| 非目标 | 说明 |
| --- | --- |
| 不做开放式全能 Copilot | 本期只覆盖投放异常、归因差异和相关知识查询，不承诺回答所有广告业务问题。 |
| 不做自动预算调整 | 不直接暂停 campaign、调整出价、修改预算、改定向、改归因配置。 |
| 不做客户自动发送 | 本期默认只输出内部诊断摘要和人工升级摘要；客户回复草稿不计入 MVP 验收，如需试点必须单独开启风控开关并人工确认。 |
| 不做 SDK/API 深度日志排障 | 本期只使用 postback 摘要和归因链路状态，不分析原始 SDK 日志、Token、设备级明细。 |
| 不做素材审核自动判定 | 素材审核与合规可作为后续阶段，不进入本期验收范围。 |
| 不用 AI 替代最终责任人 | AI 负责辅助诊断和证据汇总，最终客户承诺、赔偿、合同解释和高风险判断由人工负责。 |
| 不把用户反馈直接用于训练 | 用户反馈先进入 Badcase 和评测治理，脱敏、审核、授权后才可作为训练或调优候选。 |

## 5. 核心原则

| 原则 | 含义 | 对方案/实现的约束 |
| --- | --- | --- |
| 证据优先 | 没有可靠证据，不输出强结论。 | 实现层必须定义 evidence object、引用规则和无证据兜底。 |
| Workflow 优先于自由推理 | 投放和归因的固定必查项由规则/workflow 保证。 | 模型负责理解、总结和排序，指标计算、权限、工具顺序、最终置信度由确定性系统校验。 |
| 只读先行 | MVP 只读查询，不执行高风险写操作。 | 工具注册表不得包含预算、出价、配置修改类工具。 |
| 人在回路 | 高风险、低置信、证据冲突、客户可见输出必须人工确认。 | 前端和输出 schema 必须支持 `needs_human_review` 与人工接管原因。 |
| 内外有别 | 内部诊断可以包含更完整证据，客户可见内容必须过滤敏感字段和不确定结论。 | evidence object 必须包含 `customer_visible_allowed`。 |
| 可评测可回滚 | 每次 Prompt、知识、路由、工具策略变化都能回归测试。 | 实现层必须定义 Golden Dataset、Badcase 回归集、版本号和上线门禁。 |
| 口径透明 | 数据差异要暴露来源、时区、归因窗口、事件定义和刷新时间。 | 诊断卡必须展示数据口径和新鲜度。 |

## 6. 范围边界

### 6.1 本期范围

| 范围 | 说明 |
| --- | --- |
| 投放效果异常诊断 | 覆盖消耗、曝光、点击、CTR、CPC、CPI、CPA、ROAS 等异常的初步拆解和排查建议。 |
| 归因与数据不一致核对 | 覆盖平台、MMP、客户后台/BI 之间安装、注册、付费等事件数据差异的核查清单和原因排序。 |
| 知识检索与引用 | 检索投放 SOP、MMP/平台文档、指标口径、归因口径、历史案例，并生成引用。 |
| 只读数据工具 | 通过授权工具读取平台报表、MMP 报表、postback 摘要和相似案例。 |
| 证据管理 | 把工具结果、文档引用、规则计算结果转成 evidence object。 |
| 内部诊断输出 | 输出可能原因、证据、已排除项、下一步动作、待补充信息和人工接管建议。 |
| 反馈与 Badcase | 支持采纳、无效、证据不足、核查遗漏、越权风险等反馈进入回归闭环。 |

### 6.2 暂不做

| 暂不做 | 后续触发条件 |
| --- | --- |
| SDK/API 原始日志级排障 | postback 摘要不足以解释常见归因差异，且技术支持可提供脱敏日志字段契约。 |
| 素材审核与合规自动诊断 | 投放/归因 MVP 指标稳定，且审核政策、行业规则和落地页检查素材可结构化。 |
| 客户多语言自动回复发送 | 内部诊断和客户回复草稿通过人工审阅，且法务/客户成功确认话术边界。 |
| 自动执行优化动作 | 只读诊断采纳率稳定，且预算、出价、暂停等动作具备审批、回滚和审计能力。 |
| 跨平台商业化 SaaS | 内部工具验证稳定，数据连接、权限隔离和租户计费模型明确。 |

### 6.3 强约束

1. 所有数据工具必须经过权限校验，只允许读取聚合数据或脱敏摘要。
2. 不允许模型生成不存在的工具、字段或来源。
3. 模型不得把相关性解释成确定因果；证据不足时必须给出待确认项。
4. 客户可见内容不得展示内部成本、利润、客户隐私、原始日志、token、设备级明细或内部责任判断。
5. 任何合同、赔偿、法律承诺、平台违规结论必须转人工。

## 7. 关键决策

| 决策 | 结论 | 依据 | 影响文档 |
| --- | --- | --- | --- |
| PRD 形态 | 使用三文件分层 PRD：决策层、方案层、实现层 | 旧 PRD 长上下文混杂，容易漂移；技能要求最终只交付三份 PRD | 全部 |
| MVP 场景 | 先做投放效果异常和归因差异核对 | README 明确第一阶段 PRD 聚焦这两类；方向文档三类场景中前两类最适合数据+知识排障 | `02-solution.md`、`03-implementation.md` |
| AI 架构 | RAG + 意图路由 + 场景 workflow + 只读工具 + 规则校验 + 人工确认 | 方向文档明确 RAG、意图分类、规则引擎、数据查询 Agent、人工确认闭环 | `03-implementation.md` |
| 工具边界 | 只读工具优先，不开放写操作 | 广告预算和客户承诺风险高，MVP 目标是可控可评测 | `03-implementation.md#7-tool-use--function-calling` |
| 证据契约 | 所有主结论绑定 evidence object | 降低幻觉和客户误导风险，支撑追溯与评测 | `03-implementation.md#8-prompt上下文与输出-schema` |
| 指标口径 | 方向材料数字仅作为估算，MVP 指标需通过灰度验证 | 方向文档明确为初步估算，不能包装成真实结果 | 本文第 3 章 |
| 客户回复 | 本期默认不生成可直接发送的客户回复，只输出内部诊断摘要和升级摘要 | 客户可见内容存在合规、承诺和责任归属风险 | `02-solution.md`、`03-implementation.md` |
| 评测优先级 | 先守风险指标，再看效率指标 | AI 排障错误会伤害客户信任，必须先控制越权和错误建议 | `03-implementation.md#11-评测体系与-mlops` |

## 8. 被拒方案

| 方案 | 被拒原因 | 后续触发条件 |
| --- | --- | --- |
| 只做通用知识库问答 | 无法处理广告指标拆解和跨系统数据核对，差异化不足 | 可作为纯知识查询子能力保留 |
| 让大模型自由规划所有排障步骤 | 归因核查必查项稳定，自由规划容易漏查、不可测、成本高 | 复杂冲突证据可升级模型解释，但必查项仍由 workflow 保证 |
| 一期覆盖 SDK、素材审核、客户回复、自动优化 | 范围过大，数据契约和评测集不足，容易重新变成长文档幻觉 | 投放/归因 MVP 稳定后按模块扩展 |
| 直接对接所有外部平台 API | 接入成本高，权限和数据口径复杂 | 先用内部聚合报表、MMP 摘要、postback 摘要和人工上传样本验证 |
| 把 AI 采纳率作为唯一成功指标 | 采纳率可能鼓励迎合用户，不能衡量证据质量和安全 | 与错误建议率、引用覆盖率、越权阻断率共同看 |
