## 第零部分：项目摘要、商业论证与项目蓝图

### 0.1 项目摘要（Executive Summary）

- 产品定位：AdOps Copilot 投放归因排障助手是面向移动互联网广告业务的一线内部 AI 排障 Copilot，帮助广告运营、客户成功、广告优化师和技术支持在一个入口完成投放效果异常、归因/数据不一致和知识口径查询的初步诊断。
- AI 系统形态：总控 Agent + 投放效果诊断 Agent + 归因核对 Agent + 知识与案例检索 Agent + RAG 知识库 + 只读数据工具 + JudgeAI/Badcase 闭环。
- V1.0 最小闭环：用户提出投放或归因问题，系统识别意图和缺失槽位，检索 SOP/归因口径/历史案例，调用授权只读工具，输出带证据对象的内部诊断结论、排查路径、下一步动作和是否升级人工。
- 目标用户：广告运营、客户成功/AM、广告优化师、技术支持，以及维护广告口径和 SOP 的产品/运营角色。
- 核心价值：把“问同事、翻文档、查多个后台、再人工组织结论”的排障过程，收敛为可追溯、可评测、可复用的 AI 辅助工作流。
- 最大风险：AI 把相关性误判为因果、引用过期或无权限知识、对客户可见内容过度承诺、以及把模拟验证指标误写成真实上线结果。

## 第一部分：产品愿景与核心原则

### 1.1 项目背景

移动互联网广告投放链路包含广告计划、素材、出价、流量、点击、安装、注册、付费、MMP 归因、平台报表、BI 报表和客户沟通。源材料明确指出，广告平台、MMP 和广告主 BI 的数据口径不一致，归因解释成本高；素材审核、SDK 接入、事件回传、广告消耗异常等问题高频发生，但知识分散在平台文档、内部 SOP、历史工单和员工经验中。

当前一线排障常见路径是：

```text
客户或内部角色提出问题
  -> AM/客户成功初步理解
  -> 广告运营查看账户和报表
  -> 数据/BI 同学核对口径
  -> 技术支持确认 SDK、postback 或 API 状态
  -> 产品/运营确认平台机制
  -> AM 再组织回复
```

这个流程的低效点不只是“查得慢”，而是上下文在多人之间不断转述：问题描述、账户、时间窗、截图、指标口径、已排查项和下一步动作容易丢失。AdOps Copilot 的机会在于把这种隐性排查经验转成结构化工作流，并把知识、数据、证据和人工反馈沉淀为可复用资产。

### 1.2 产品目标与核心价值

| 目标 | 用户价值 | 业务价值 | AI 能力支撑 |
| --- | --- | --- | --- |
| 建立投放归因排障第一入口 | 一线不必先判断查哪个系统或问谁 | 缩短问题定位链路，减少重复沟通 | 总控 Agent、Intent 路由、槽位追问 |
| 输出证据型诊断 | 运营能看到可能原因、依据、已排除项和下一步动作 | 提高诊断一致性，减少返工 | RAG、只读工具、evidence object |
| 固化资深人员排查逻辑 | 新人按指标链路和归因核查清单处理问题 | 降低团队对个别专家的依赖 | 投放诊断 Agent、归因核对 Agent、规则引擎 |
| 沉淀历史案例与 Badcase | 类似问题可以复用，错误可追溯修复 | 形成知识资产和质量飞轮 | 相似案例检索、Badcase 队列、回归集 |
| 控制 AI 风险 | 无证据、越权、对外承诺被拦截 | 降低数据、合规和商务风险 | Permission Guard、Delivery Guard、JudgeAI |

### 1.3 范围与边界（V1.0）

#### 1.3.1 V1.0 做什么

| 范围 | 说明 | 交付结果 |
| --- | --- | --- |
| 投放效果异常诊断 | 面向消耗、曝光、点击、安装、注册、付费、cvr、cpa、roi 等指标波动 | 指标拆解、可能原因排序、证据卡、下一步检查 |
| 归因与数据不一致核对 | 面向平台、MMP、BI 或客户后台之间的安装/转化差异 | 时区、归因窗口、事件映射、去重、postback、数据延迟核查 |
| 知识口径查询 | 面向投放 SOP、指标定义、MMP 口径、归因规则、历史案例 | 带引用的解释、适用范围、限制说明 |
| 内部诊断摘要 | 面向 AM/客户成功的内部可转述要点 | 不直接发送客户，仅辅助人工组织沟通 |
| Badcase 与评测闭环 | 面向 AI 输出质量治理 | Badcase 标签、回归集、SME 抽检和修复状态 |
| 只读工具查询 | 查询平台报表、MMP 报表、postback 摘要和知识库 | 工具结果归一为 evidence object |

#### 1.3.2 V1.0 绝不做什么

| 不做 | 不做原因 | 用户替代路径 |
| --- | --- | --- |
| 自动调整预算、出价、定向、素材状态 | 直接影响广告花费和客户投放结果，风险高 | 输出建议后由有权限人员人工执行 |
| 自动发送客户回复 | 对外承诺、商务风险和合规风险高 | 输出内部摘要和可转述要点，客户成功人工编辑确认 |
| 深度 SDK/API 日志排障 | 涉及 raw log、设备级数据、错误码和工程修复链路，权限与脱敏复杂 | V1 仅查询 postback 摘要；深度排障进入 V1.1+ |
| 多模态素材审核 | 涉及图片/OCR/落地页/政策判断，误判成本高 | 作为后续独立 Agent 评估，不阻塞投放归因 MVP |
| 对客户开放独立入口 | 内部质量、权限和合规尚未经过足够灰度 | 先面向内部 AdOps/AM 使用 |
| 把模拟指标写成真实收益 | 项目尚未真实上线，不能编造业务结果 | 使用 PoC 目标、测算模型和待验证标记 |

### 1.4 核心原则

| 原则 | 含义 | 对 Agent/RAG/Tool/Eval 的约束 |
| --- | --- | --- |
| 循证优先 | 诊断必须说明依据来源，不能只给经验判断 | 每个主结论至少绑定 1 个 evidence object；无证据输出待确认或转人工 |
| 只读安全 | V1 只做查询、诊断、摘要，不做账户变更 | 工具注册表只允许 `get_platform_report`、`get_mmp_report`、`get_postback_summary`、`search_knowledge_base`、`search_similar_cases` |
| 人在回路 | 高风险、低置信、对外表达和权限边界必须人工确认 | `human_review_required` 是交付状态，不是失败 |
| 规则兜底 | 指标公式、权限、风险终判、状态机不可交给模型自由决定 | LLM 只负责候选理解和证据解释，规则层负责最终约束 |
| 可评测上线 | 每次 Prompt、知识、工具和模型变更都必须可回归 | Golden Set、JudgeAI Set、Regression Badcase Set 进入门禁 |
| 成本可控 | AI 能力按场景分层，不用强模型解决所有问题 | 路由、缓存、rerank、工具超时和高成本配额写入实现层 |
| 事实/推断分离 | 文档和输出区分事实、可能原因、建议和待验证 | 回答 Schema 必须包含 confidence、risk_flags、evidence_ids 和 limitations |
