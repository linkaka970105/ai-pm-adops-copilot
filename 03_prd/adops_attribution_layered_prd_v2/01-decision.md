# 决策层：AdOps Copilot 投放归因排障助手 v2

## 来源链接

- PRD 状态：[00-prd-state.md](00-prd-state.md)

## 0. 决策摘要

AdOps Copilot 投放归因排障助手的第一阶段目标，是为广告运营、客户成功、广告优化师和技术支持提供一个内部排障入口，优先解决两类高频问题：投放效果异常诊断，以及广告平台、MMP、客户 BI 之间的归因和数据不一致核对。

本版不做全能 AdOps Copilot，不覆盖 SDK/API 深排、素材审核、客户可直接发送回复、预算/出价/账户配置变更。系统定位是“证据化的内部诊断助手”：先把用户问题结构化，检索知识与历史案例，调用授权只读工具，输出可追溯的诊断结论、核查清单、下一步动作和人工升级摘要。

关键决策：

| 决策 | 内容 | 证据 |
|---|---|---|
| MVP 范围 | 投放效果异常诊断、归因数据不一致核对、知识查询、证据管理、Badcase 回流 | E-001, E-009 |
| AI 形态 | RAG + 意图路由 + 指标规则 + 只读工具调用 + 相似案例检索 + 人工确认闭环 | E-008, E-010 |
| 安全边界 | 第一阶段只做内部诊断和建议，不自动执行高风险动作，不生成可直接对客户发送的最终回复 | E-010, E-013 |
| 可信输出 | 所有业务结论必须绑定 Evidence Object；无可靠证据时输出缺口、追问或转人工 | E-015, E-016 |
| 指标口径 | 收益和效率提升先作为目标假设；上线前必须采集 2-4 周人工基线 | E-007, E-020 |

## 1. Why - 为什么做

### 1.1 业务背景

移动互联网广告链路横跨广告平台、MMP、客户 BI、SDK、postback、素材审核和客户服务流程。投放异常和归因差异常见但难排查，原因通常不是单点故障，而是时间窗口、统计口径、转化链路、素材、预算、流量结构、事件映射、postback 延迟等因素叠加。

从已有方向材料看，当前团队面临四类典型问题：

| 问题 | 说明 | 证据类型 |
|---|---|---|
| 知识分散 | SOP、平台文档、MMP 文档、历史工单、IM 记录、BI 看板分散，用户不知道先查哪里 | E-005，事实/用户洞察 |
| 依赖资深经验 | CPA、CPI、ROAS、安装数、postback 等异常需要按指标链路拆解，新人难以判断优先核查项 | E-005，事实/用户洞察 |
| 跨团队流转慢 | AM、运营、数据、技术支持、产品和外部平台之间反复补充上下文，问题流转成本高 | E-005，事实/用户洞察 |
| 解释不稳定 | 同类数据差异问题可能被不同人用不同口径解释，缺少标准化证据和核查路径 | E-005，事实/用户洞察 |

### 1.2 为什么需要 AI

传统 FAQ、固定 BI 看板和规则告警只能解决“是什么”和“哪里异常”的一部分问题，难以处理真实排障中的动态组合：

- 用户输入通常是自然语言，且可能缺少 account、campaign、app、event、time range、timezone 等关键字段。
- 排障需要同时理解广告指标、归因口径、平台配置、MMP 规则、历史案例和工具返回数据。
- 诊断结论必须能解释“为什么可能是这个原因”，并说明还缺哪些证据。
- 一线团队需要内部诊断摘要、下一步核查动作和可升级给技术/数据团队的结构化上下文。

因此本产品采用 RAG、意图路由、指标规则、只读工具调用和人工确认闭环，而不是单纯做 FAQ 搜索或开放式聊天。

### 1.3 业务目标

| 维度 | 现状 | 目标 | 如何衡量 | 证据/状态 |
|---|---|---|---|---|
| 排障效率 | 人工跨系统查文档、问同事、查后台和整理结论 | 降低投放/归因问题的首轮定位时间 | 人工基线 P50 处理时长 vs AI 辅助 P50 处理时长 | E-005, E-007；需基线 |
| 诊断质量 | 结论依赖个人经验，证据和口径不稳定 | 输出带证据、口径、下一步动作的内部诊断 | 采纳率、返工率、证据引用覆盖率、人工抽检通过率 | E-005, E-015 |
| 协作效率 | 升级技术/数据团队时上下文不完整 | 自动生成结构化升级摘要和缺失字段清单 | 升级补充次数、重复沟通次数、升级有效率 | E-005；需工单数据 |
| 经验沉淀 | 错误回答和历史处理经验难沉淀 | Badcase 回流到知识库、Prompt、工具和评测集 | Badcase 修复周期、回归通过率、有效知识新增数 | E-008, E-017 |

### 1.4 成功标准

以下是第一阶段建议目标。除“越权率 0”和“高风险自动执行 0”属于硬门槛外，效率和收益类目标必须在灰度前完成基线采集，不能作为已达成结果使用。

| 指标类型 | 指标名称 | 当前值 | 目标值 | 观测周期 | 说明 |
|---|---|---|---|---|---|
| 北极星指标 | 有效 AI 辅助投放归因排障会话数 | 待采集 | 灰度期连续 4 周增长，且 Badcase 率不升高 | 周/月 | 会话需真实、授权、带证据、有采纳或后续动作 |
| 业务效果 | AI 辅助解决率 | 待采集 | 灰度期目标 50% 以上 | 周 | 指无需升级即可形成可执行下一步动作的会话比例 |
| 用户采纳 | 人工采纳率 | 待采集 | 小流量灰度目标 60% 以上 | 周 | 不与 AI 辅助解决率混用 |
| 质量 | 诊断返工率 | 待采集 | 灰度期逐周下降 | 周 | 由人工抽检和 Badcase 标记确认 |
| 证据可靠性 | 关键结论证据绑定率 | 待采集 | P0 场景 100% | 日/周 | 无 evidence 不允许输出确定性业务结论 |
| 安全 | 越权工具调用率 | 待采集 | 0 | 实时 | 工具网关硬拦截 |
| 安全 | 高风险自动执行率 | 待采集 | 0 | 实时 | 本版不自动改预算、出价、配置或发送客户回复 |
| 运营 | 核心 Badcase 修复周期 | 待采集 | 建议小于 7 个工作日 | 周 | 需确认 Owner 与 SLA |

### 1.5 不做的后果

| 影响周期 | 后果 | 说明 |
|---|---|---|
| 短期 | 一线排障继续依赖人工经验 | 新人需要反复问资深同事，重复问题消耗技术和运营时间 |
| 中期 | 诊断和客户解释口径不稳定 | 同类归因差异可能被不同团队用不同语言解释，影响客户信任 |
| 长期 | 经验难以转为数据资产 | 历史工单、SOP、口径修订和 Badcase 不能持续沉淀，模型和流程无法迭代 |
| 竞争 | 平台型或 MMP 型 AI 助手会占据用户心智 | AppsFlyer、Optmyzr 等竞品已验证自然语言数据分析和根因解释的价值 |

### 2.2 典型使用场景

| 场景 | 用户 | 触发条件 | 当前痛点 | 期望结果 | 优先级 |
|---|---|---|---|---|---|
| 投放效果异常诊断 | 广告运营、广告优化师 | 消耗下降、CPA/CPI 上升、ROAS 下降、安装或转化异常 | 需要跨 campaign、geo、OS、素材、广告位和漏斗指标拆解 | 输出异常维度、可能原因排序、证据和下一步核查动作 | P0 |
| 归因与数据不一致核对 | 客户成功、广告运营、技术支持 | 平台、MMP、客户 BI 对安装/注册/付费等数据不一致 | 时区、窗口、事件定义、postback、去重、SKAN/SAN 口径复杂 | 输出差异解释、核查清单、缺失字段、是否需要技术介入 | P0 |
| 纯知识查询 | 广告运营、客户成功 | 用户问指标定义、归因窗口、postback 机制、SOP | 关键词搜索难以命中口径，答案缺少引用 | 输出带来源引用的解释；无可靠引用时拒绝确定回答 | P0 |

| Badcase 复盘 | 知识库 Owner、算法、产品 | 用户点踩、证据冲突、工具失败、人工标记错误 | 错误样本散落，不能转为回归数据 | 创建 Badcase，标注根因，进入修复和回归流程 | P1 |

### 2.3 用户旅程

```text
用户提出投放或归因问题
  -> 系统识别意图和语言
  -> 检查权限、账户范围和缺失字段
  -> 低风险知识类问题走 RAG 引用回答
  -> 投放/归因问题进入固定 workflow
  -> 调用授权只读工具和知识检索
  -> 组装 Evidence Object
  -> 输出内部诊断卡、核查清单、下一步动作和限制说明
  -> 用户采纳、继续追问、补充字段、升级人工或标记 Badcase
  -> 反馈进入评测、知识库和 Badcase 闭环
```

关键痛点出现在三个位置：

| 环节 | 痛点 | PRD 对应能力 |
|---|---|---|
| 问题进入系统前 | 用户问题不完整，不知道需要哪些字段 | 意图识别、Slot Filling、缺失字段追问 |
| 诊断过程中 | 数据源和知识源分散，结论不可追溯 | RAG、只读工具、Evidence Object |
| 诊断后 | 错误和修正无法沉淀 | 反馈、Badcase、回归数据集 |

## 3. What - 做什么

### 3.1 功能清单

| 功能 | 描述 | 优先级 | 理由 | 证据 |
|---|---|---|---|---|
| 总控意图路由 | 识别投放诊断、归因核对、知识查询、范围外和高风险请求 | P0 | 控制 workflow 和安全边界 | E-008, E-017 |
| 权限与字段校验 | 检查用户是否有账户/MMP/postback/知识访问权限，识别缺失字段 | P0 | 防越权、防无证据回答 | E-010, E-016 |
| 投放效果异常诊断 | 按指标链路和维度拆解异常，输出原因排序和证据 | P0 | 第一阶段核心业务价值 | E-009 |
| 归因与数据不一致核对 | 检查时区、窗口、事件定义、postback、去重、SKAN/SAN 等核查项 | P0 | 第一阶段核心业务价值 | E-009 |
| 知识检索与引用回答 | 检索 SOP、指标口径、归因规则和历史案例，带引用回答 | P0 | 降低知识查找成本和幻觉 | E-005, E-008 |
| Evidence Object 管理 | 统一封装知识、工具和历史案例证据，支持可见性和审计 | P0 | 让诊断可追溯、可复核 | E-015 |
| 内部诊断卡 | 展示结论、证据、置信度、限制、下一步动作和是否需人工确认 | P0 | 让一线可用、可执行 | E-016 |
| Badcase 回流 | 记录错误、证据不足、工具失败、用户点踩和人工修正 | P1 | 支撑持续优化 | E-008, E-017 |
| 结构化升级摘要 | 当无法闭环时，生成内部升级上下文 | P1 | 降低跨团队沟通成本 | E-005 |
| 管理/运营后台 | 管理知识、工具、Badcase、评测集和灰度配置 | P1 | 支撑可运营上线 | E-017 |

### 3.2 MVP 范围

第一版 MVP：

1. 入口：投放系统后台内嵌 Copilot 面板，支持从账户/Campaign 上下文进入，也支持独立输入问题。
2. 场景：投放效果异常诊断、归因与数据不一致核对、纯知识查询。
3. 工具：只读工具 `get_platform_report`、`get_mmp_report`、`get_postback_summary`、`search_knowledge_base`、`search_similar_cases`。
4. 输出：内部诊断卡、归因核查卡、证据引用卡、缺失字段追问、人工升级摘要。
5. 安全：权限前置、证据绑定、交付状态枚举、无可靠引用拒绝确定回答、高风险请求拒绝或转人工。
6. 运营：用户反馈、Badcase 标记、核心 trace 字段、黄金数据集和灰度看板。

后续迭代：

| 阶段 | 方向 | 进入条件 |
|---|---|---|
| P1 | 更完整的投放优化建议、自动生成升级工单草稿、管理后台增强 | P0 采纳率和安全指标达标 |
| P2 | SDK/API 深排、素材审核、客户可见回复、预算/出价变更建议 | 独立安全评审和数据接入完成 |
| P3 | 半自动执行低风险修复建议 | 工具权限、回滚、审计和人工确认机制成熟 |

### 3.3 不做什么

| 不做的功能 | 为什么不做 | 什么时候重新考虑 |
|---|---|---|
| 自动修改预算、出价、定向、账户配置 | 直接影响广告消耗和客户利益，风险高 | 低风险建议长期通过人工采纳，且有回滚和审批能力 |
| 自动发送客户可见回复 | 涉及客户承诺、敏感数据和语气合规，本版只做内部诊断 | 客户回复规范、人工审核流和可见性字段成熟后 |
| SDK/API 深层日志排障 | 需要 raw log、错误码、SDK 版本和技术支持流程，数据权限更复杂 | 独立 SDK/API 排障 PRD |
| 素材审核与合规判断 | 涉及多模态、政策、行业资质和人工审核队列 | 独立素材审核 PRD |
| 合同、账单、赔偿和法律承诺 | 高风险非排障场景 | 不纳入本产品；固定拒答/转人工 |
| 把示例 ROI 当真实收益 | 当前缺少真实灰度数据 | 完成基线和灰度后再更新为真实结果 |

### 3.4 决策层验收

| 验收项 | 标准 |
|---|---|
| 范围清晰 | P0 只包含投放诊断、归因核对、知识查询、证据管理和 Badcase 基础闭环 |
| 事实与假设分离 | 效率提升、节省成本、采纳率目标均标为目标或待验证，不写成已达成 |
| 安全边界明确 | 不自动执行高风险动作，不生成可直接发送客户回复 |
| 指标可观测 | 所有成功指标有数据来源或待确认问题 |
| AI 价值清楚 | 说明 AI 解决的是多源信息整合、隐性经验显性化、证据化诊断和持续回归 |
