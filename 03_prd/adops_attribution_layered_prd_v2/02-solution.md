# 方案层：AdOps Copilot 投放归因排障助手 v2

## 来源链接

- PRD 状态：[00-prd-state.md](00-prd-state.md)
- 决策层：[01-decision.md](01-decision.md)

## 0. 方案摘要

本版推荐方案为“投放系统内嵌入口 + 总控路由 + 固定场景 workflow + RAG + 只读工具 + Evidence Object + 人工确认闭环”。

该方案不追求第一阶段覆盖全部 AdOps 场景，而是先把投放效果异常和归因数据不一致两类高频问题做成可证据化、可灰度、可评测、可回滚的内部诊断工具。

## 1. Where - 功能入口

### 1.1 入口方案对比

| 方案 | 描述 | 优点 | 缺点 | 预期影响 |
|---|---|---|---|---|
| A. 投放系统后台内嵌 Copilot 面板 | 在账户、Campaign、报表和归因对账页面提供侧边栏入口，自动带入上下文 | 离数据最近，可减少用户补字段；适合投放诊断和归因核对 | 依赖投放系统改造；入口受系统权限影响 | 推荐作为 P0 主入口 |
| B. 工单系统/飞书机器人入口 | 用户从 IM 或工单发起问题，Copilot 追问字段并生成升级摘要 | 贴近客户成功和跨团队协作；上线快 | 缺少页面上下文，字段追问成本高；权限处理复杂 | 作为 P1 辅助入口 |
| C. 独立 Web 工作台 | 独立登录后输入问题、选择账户和场景 | 可快速验证 AI 诊断能力，不绑现有系统 | 用户需要切换系统；难自动读取上下文 | 可作为内部灰度/测试后台 |
| D. 报表看板中的 Analyze with AI | 在异常图表旁直接触发解释 | 对标 AppsFlyer 图表分析体验，场景清晰 | 对归因核对和跨系统诊断覆盖有限 | P1/P2 增强体验 |

推荐入口：方案 A + 方案 C。

推荐理由：

1. P0 目标是验证真实投放/归因排障价值，必须靠近账户、Campaign、指标和时间范围。
2. 独立 Web 工作台适合灰度期内部验证、Badcase 复盘和测试用例演示。
3. IM/工单入口和图表入口有价值，但容易让本版扩大到客户回复、跨团队工单和可视化分析，不作为 P0 必要条件。

### 1.2 入口状态

| 入口状态 | 触发条件 | 系统行为 |
|---|---|---|
| 带上下文进入 | 用户从账户/Campaign/报表页打开 Copilot | 自动预填 account、campaign、app、time range、metric 等可用字段 |
| 无上下文进入 | 用户从独立工作台输入问题 | 总控先识别意图和缺字段，再追问 |
| 权限不足 | 用户无账户、MMP 或 postback 摘要权限 | 不调用受限工具，展示 `permission_blocked` |
| 范围外 | 用户请求 SDK 深排、素材审核、客户回复、预算变更等 | 展示 `out_of_scope`，可记录后续需求 |

| 2. 核心 workflow | 第 6-9 周 | 投放诊断 workflow、归因核对 workflow、知识查询、内部诊断卡、工具网关 | 研发、算法、测试 | 待启动 |
| 3. 评测与灰度 | 第 10-12 周 | 黄金数据集、E2E 回归、灰度配置、监控看板、Badcase 标注队列 | 测试、算法、运营 | 待启动 |
| 4. 小流量上线 | 第 13-16 周 | 5-10 名资深 AdOps/AM 灰度、问题复盘、迭代清单、上线/回滚评审 | 产品、运营、研发 | 待启动 |

缓冲建议：每阶段保留 20% 时间处理数据接入、权限、知识质量和模型输出不稳定问题。

## 3. 灰度计划

| 阶段 | 时间 | 用户范围 | 目标 | 决策标准 |
|---|---|---|---|---|
| 离线验证 | 第 10 周 | 内部测试集 | 验证路由、RAG、工具、Evidence 和交付状态 | P0 阻塞 bug 清零；越权拦截通过 |
| 小流量灰度 | 第 13-14 周 | 5-10 名资深 AdOps/AM，限定 2-3 个账户或样本客户 | 验证真实问题采纳率和诊断质量 | 人工采纳率达到 60% 以上；越权率 0；严重 Badcase 可闭环 |
| 场景扩展灰度 | 第 15 周 | 增加投放诊断和归因核对样本 | 验证跨场景稳定性 | AI 辅助解决率达到目标区间，诊断返工率不升高 |
| 扩大试点 | 第 16 周后 | 更多 AdOps/AM/技术支持 | 验证运营流程和知识迭代 | Badcase 修复 SLA 可执行，工具失败率和延迟可接受 |

观测指标：

| 类型 | 指标 | 门槛/目标 |
|---|---|---|
| 稳定性 | 系统错误率、工具超时率、P95 延迟 | 需在灰度前由研发/测试确认阈值 |
| 安全 | 越权工具调用率、高风险自动执行率 | 均为 0 |
| 效果 | 人工采纳率 | 小流量灰度目标 60% 以上 |
| 效果 | AI 辅助解决率 | 灰度期目标 50% 以上，需与采纳率区分 |
| 质量 | 关键结论证据绑定率 | P0 场景 100% |
| 运营 | 核心 Badcase 修复周期 | 建议小于 7 个工作日，待确认 |

## 4. How - 解决方案路线对比

### 4.1 方案 A：纯 FAQ/规则库

- 描述：把常见投放和归因问题整理为 FAQ，用户搜索后查看答案。
- 优点：成本低、上线快、风险低。
- 缺点：无法处理自然语言复杂问题，不能跨报表和 MMP 数据取证，无法输出个性化诊断。
- 成本：低。
- 周期：2-4 周。
- 结论：可作为知识库底座，不足以成为排障助手。

### 4.2 方案 B：RAG 知识问答

- 描述：用户输入问题，系统从 SOP、MMP 文档、指标口径、历史工单中检索并生成带引用回答。
- 优点：适合知识分散问题，能降低搜索成本。
- 缺点：对投放数据异常和归因差异缺少工具取证；容易停留在口径解释。
- 成本：中。
- 周期：4-6 周。
- 结论：适合知识查询 P0，但不能单独覆盖诊断场景。

### 4.3 方案 C：RAG + 总控路由 + 固定 workflow + 只读工具

- 描述：总控识别意图和权限后，将问题路由到投放诊断、归因核对或知识查询 workflow；workflow 调用 RAG 和只读数据工具，形成 Evidence Object 和内部诊断卡。
- 优点：在 AI 灵活性和工程可控性之间平衡；能证据化输出；适合灰度和回归评测。
- 缺点：需要工具网关、权限模型、schema 校验和知识治理；初期覆盖范围有限。
- 成本：中高。
- 周期：10-16 周。
- 结论：推荐作为本版主方案。

### 4.4 方案 D：多 Agent 自动排障与执行

- 描述：多个智能体自动规划、调用工具、修改配置、生成客户回复并跟进工单。
- 优点：长期自动化潜力大。
- 缺点：第一阶段风险过高；容易越权、误执行或输出不可控客户承诺；评测难度高。
- 成本：高。
- 周期：16 周以上。
- 结论：不作为本版 MVP，仅保留为后续演进方向。

### 4.5 推荐方案

推荐方案：方案 C。

推荐理由：

1. 符合现有材料中的 AI 技术路线：RAG、意图分类、规则引擎、数据查询 Agent、相似案例检索和人工确认闭环。
2. 能把竞品中的优势拆开吸收：AppsFlyer 的自然语言营销数据分析、Optmyzr 的指标拆解、Zendesk 的工单/知识闭环。
3. 能清楚控制风险：只读工具、权限前置、证据绑定、无证据拒答、范围外转人工。
4. 能评测和灰度：每个 workflow 有输入、输出、固定核查项、Evidence Object 和 Badcase 回流。

## 5. 架构路线

```text
用户入口
  -> 会话与上下文层
  -> 总控路由层
      -> 意图识别
      -> 权限校验
      -> 缺字段追问
      -> Query rewrite / 检索计划
  -> 场景 workflow 层
      -> 投放效果异常诊断
      -> 归因与数据不一致核对
      -> 知识查询
  -> 信息供给层
      -> RAG 知识检索
      -> get_platform_report
      -> get_mmp_report
      -> get_postback_summary
      -> search_similar_cases
  -> Evidence Builder
  -> Delivery Guard
  -> 内部诊断卡 / 归因核查卡 / 知识回答 / 升级摘要
  -> 用户反馈 / Badcase / 回归数据集
```

关键职责边界：

| 层级 | 负责什么 | 不负责什么 |
|---|---|---|
| 总控路由层 | 意图、权限、缺字段、风险、workflow 选择、检索/工具计划 | 不生成最终业务结论 |
| 场景 workflow | 固定核查步骤、工具调用顺序、证据解释、原因排序 | 不绕过权限、不自由新增工具 |
| Evidence Builder | 统一证据对象、来源、可见性、Owner、检索时间 | 不决定最终展示文案 |
| Delivery Guard | 根据证据、权限、风险和状态决定可展示内容 | 不重新诊断业务原因 |
| Judge AI | 离线评测、灰度回归、人工抽检、Badcase 复盘 | 不作为线上每次回答的强制拦截器 |

## 6. 风险管理

| 风险 | 发生概率 | 影响程度 | 应对方案 | 负责人 |
|---|---|---|---|---|
| 范围膨胀为全能 AdOps Copilot | 高 | 高 | 严格限定 P0；范围外请求走 `out_of_scope`；后续需求单独 PRD | 产品 |
| 工具或权限未准备好 | 中 | 高 | 灰度前完成 API 可行性确认；工具不可用时输出 `tool_degraded` 或追问/转人工 | 研发/数据 |
| RAG 引用不准或知识过期 | 高 | 高 | 知识源 Owner、更新时间、rerank、冲突检测、无可靠引用拒答 | 知识库 Owner/算法 |
| 诊断结论被用户误当成确定因果 | 中 | 高 | 输出“可能原因排序”和证据限制，不把相关性写成因果；关键结论人工确认 | 产品/算法 |
| 旧工具名和旧字段混入新版 | 中 | 中 | 工具名统一为五个只读工具；Evidence 字段统一用 `customer_visible_allowed` | 产品/研发 |
| 灰度指标被混用 | 中 | 中 | 人工采纳率 60% 与 AI 辅助解决率 50% 分开定义 | 产品/运营 |
| 评测覆盖被过度外推 | 中 | 中 | 明确当前 E2E 只覆盖 6 个路由 case 和 1 个归因诊断 case；其他模块待扩展 | 测试/算法 |
| 成本和收益假设被当成真实结果 | 中 | 高 | 只保留公式和口径；真实收益需基线和灰度数据 | 产品/运营 |

## 7. How much - 成本与 ROI

### 7.1 人天估算

以下为第一阶段规划估算，用于排期讨论，不代表已投入成本。

| 角色 | 工作内容 | 人天 | 备注 |
|---|---|---:|---|
| 产品经理 | 范围定义、流程设计、指标口径、验收、灰度复盘 | 20-25 | 含评审和运营规则 |
| 后端研发 | 工具网关、权限、Evidence Store、workflow API、日志埋点 | 35-45 | 取决于现有 API 复用度 |
| 前端研发 | Copilot 面板、诊断卡、证据卡、异常态、反馈入口 | 20-30 | P0 以后台内嵌入口为主 |
| 算法/AI 工程 | RAG、路由、Prompt/输出 schema、评测脚本、模型对比 | 30-40 | 不含大规模模型训练 |
| 数据工程 | 平台/MMP/postback 聚合数据接入、基线报表、监控 | 20-30 | API 未就绪时上浮 |
| 测试 | 功能测试、安全测试、E2E 回归、灰度监控 | 15-25 | 包含 golden dataset 维护 |
| 运营/知识库 Owner | SOP 清洗、口径确认、Badcase 复核、知识更新 | 15-25 | 灰度期持续投入 |

### 7.2 ROI 计算口径

本版不预设真实 ROI。上线前先采集 2-4 周人工基线，灰度后按同类问题对比。

| 收益项 | 计算口径 | 数据来源 | 注意事项 |
|---|---|---|---|
| 人效节省小时 | `有效会话数 × max(人工基线P50处理分钟 - AI辅助P50处理分钟, 0) / 60` | 工单、Copilot Trace、人工基线表 | 剔除测试、无权限、范围外和无证据会话 |
| 响应改善 | `人工首次有效响应P50 - Copilot首版可用诊断P50` | 工单、会话日志 | 首版可用诊断必须包含 evidence |
| 升级减少 | `人工基线升级率 - AI辅助后升级率` | 工单状态、升级摘要 | 只统计同类投放/归因问题 |
| 诊断质量提升 | `1 - 返工会话数 / 已交付诊断会话数` | Badcase、人工抽检 | 按场景分层抽样 |
| 知识资产沉淀 | 有效知识新增数、Badcase 关闭数、回归集新增数 | 知识库、评测后台 | 只有 Owner 审核通过才计入 |

### 7.3 决策层 ROI 边界

| 可写入 PRD | 不应写入 PRD |
|---|---|
| 目标、公式、埋点、基线采集计划、灰度验证方法 | “已节省 50% 时间”“已提升 60% 效率”等未验证结果 |
| 示例估算，且明确标注为假设 | 简历或汇报中把示例估算当成真实上线结果 |
| 分场景 P50 对比 | 混合 SDK、素材审核、客户回复等非本版场景的数据 |

## 8. 方案层验收

| 验收项 | 标准 |
|---|---|
| 推荐入口清晰 | P0 为投放系统内嵌入口 + 独立灰度工作台 |
| 推荐方案清晰 | 选择 RAG + 总控路由 + 固定 workflow + 只读工具 |
| 灰度门槛一致 | 人工采纳率用 60%，AI 辅助解决率用 50%，两者不混用 |
| 风险有 Owner | 范围、权限、知识、工具、评测和 ROI 风险均有应对 |
| 成本不冒充事实 | 人天和 ROI 均为估算口径，真实收益待基线/灰度验证 |
